02
Text Chunking

Splitting Reviews Into Embeddable Chunks

Before embedding, long reviews must be split into chunks that fit within the model's token limit. Get chunking wrong and you embed truncated text — your vectors mean nothing.

📰
The Newspaper Analogy A newspaper article is too long to summarise in one sentence. But each paragraph can be summarised individually. Chunking is exactly this — break the long review into paragraphs (chunks), embed each paragraph, and now you can find the exact paragraph about "cold naan" even inside a 500-word review.
🔥
Production Issue #1: Token Limit Silent Truncation text-embedding-3-small has an 8,191-token limit. If you send a 12,000-token review, the OpenAI API silently truncates it to 8,191 tokens and returns a vector for only part of the text. You get no error. Your vector is now semantically wrong. Fix: always count tokens BEFORE calling the embedding API and split proactively.
⚠️
Production Issue #2: Chunk Overlap vs. Storage Cost Tradeoff Overlap ensures a sentence that straddles two chunks is not lost. But 20% overlap on 50K reviews × 200 tokens/chunk = ~20% more embedding API cost and ~20% more Qdrant storage. For short reviews (average 120 tokens), overlap is often wasteful. Profile your data first — most curry bowl reviews are <200 tokens, so we use 50-token overlap, not 200.
pythonrag/chunker.py
"""
Review text chunker — splits reviews into token-safe chunks for embedding.
Key principle: a chunk must be semantically self-contained (a reader should
understand it without needing the surrounding text).
"""
from __future__ import annotations
from dataclasses import dataclass
from typing import Iterator
import tiktoken
import structlog

log = structlog.get_logger(__name__)

# text-embedding-3-small uses cl100k_base tokeniser
TOKENISER     = tiktoken.get_encoding("cl100k_base")
MAX_TOKENS    = 512   # well under the 8191 limit — gives headroom for metadata
OVERLAP_TOKENS= 50    # 50-token overlap for short review corpus

@dataclass
class ReviewChunk:
    chunk_id:       str    # "{event_id}_chunk_{index}"
    event_id:       str
    chunk_index:    int
    total_chunks:   int
    text:           str
    token_count:    int
    # Metadata carried through to Qdrant payload
    location_id:    str
    source_platform:str
    star_rating:    float
    review_date:    str

def count_tokens(text: str) -> int:
    return len(TOKENISER.encode(text))

def chunk_review(
    event_id: str,
    text: str,
    metadata: dict,
    max_tokens: int = MAX_TOKENS,
    overlap: int = OVERLAP_TOKENS,
) -> list[ReviewChunk]:
    """
    Split a single review into chunks.
    
    Short reviews (< max_tokens): one chunk, no splitting.
    Long reviews: sliding window with overlap.
    
    IMPORTANT: overlap is in TOKENS, not characters.
    "100 character overlap" is meaningless — token boundaries don't
    map 1:1 to characters. Always count in tokens.
    """
    token_ids = TOKENISER.encode(text)
    total_tokens = len(token_ids)

    # Fast path: entire review fits in one chunk
    if total_tokens <= max_tokens:
        chunk_text = text.strip()
        return [ReviewChunk(
            chunk_id        = f"{event_id}_chunk_0",
            event_id        = event_id,
            chunk_index     = 0,
            total_chunks    = 1,
            text            = chunk_text,
            token_count     = total_tokens,
            **metadata
        )]

    # Sliding window chunking
    chunks: list[ReviewChunk] = []
    start  = 0
    index  = 0
    stride = max_tokens - overlap   # how far we advance each window

    while start < total_tokens:
        end        = min(start + max_tokens, total_tokens)
        chunk_ids  = token_ids[start:end]
        chunk_text = TOKENISER.decode(chunk_ids).strip()

        if chunk_text:   # skip empty chunks from decode artifacts
            chunks.append(ReviewChunk(
                chunk_id        = f"{event_id}_chunk_{index}",
                event_id        = event_id,
                chunk_index     = index,
                total_chunks    = -1,   # patched after loop
                text            = chunk_text,
                token_count     = len(chunk_ids),
                **metadata
            ))

        start += stride
        index += 1

    # Patch total_chunks now that we know it
    for c in chunks:
        c.total_chunks = len(chunks)

    log.debug("review_chunked",
               event_id=event_id,
               total_tokens=total_tokens,
               num_chunks=len(chunks))
    return chunks


def chunk_batch(reviews: list[dict]) -> Iterator[ReviewChunk]:
    """Yield chunks for a batch of review dicts from Gold layer."""
    for review in reviews:
        metadata = {
            "location_id":     review["location_id"],
            "source_platform": review["source_platform"],
            "star_rating":     float(review["star_rating"]),
            "review_date":     str(review["review_date"]),
        }
        yield from chunk_review(
            event_id = review["event_id"],
            text     = review["clean_text"],
            metadata = metadata,
        )

03
Embeddings

Embedding Reviews with OpenAI text-embedding-3-small

Convert every text chunk into a 1536-dimensional numeric vector. This is the heart of semantic search. The pipeline must handle API rate limits, batch sizing, cost tracking, and retry — all without losing a single chunk.

🗺️
The GPS Coordinate Analogy Just as a city can be described by two numbers (latitude, longitude) and similar cities end up geographically close, an embedding describes a piece of text by 1536 numbers. Similar texts end up "close" in this 1536-dimensional space. "Slow service" and "waited forever" are neighbours. "Great naan" and "perfect biryani" are neighbours. "Slow service" and "great naan" are far apart. The embedding model is the GPS system that assigns these coordinates.
🔥
Production Issue #3: OpenAI Rate Limits Killing Bulk Indexing OpenAI's default rate limit is 3,000 RPM (requests per minute) and 1,000,000 TPM (tokens per minute). Naively embedding 50K reviews in a tight loop hits the TPM limit within 30 seconds — you get a 429, your script crashes, and you have to start over. Fix: batch up to 100 chunks per API call (OpenAI supports array input), track tokens-per-minute in a sliding window, and sleep when approaching the limit.
pythonrag/embedder.py
"""
Embedding pipeline: ReviewChunk → (chunk, vector) pairs.
Handles OpenAI rate limits, batching, cost tracking, and retries.
"""
from __future__ import annotations
import time, os
from typing import Iterator
import structlog
from openai import OpenAI
from tenacity import (
    retry, stop_after_attempt, wait_exponential_jitter,
    retry_if_exception_type
)
from openai import RateLimitError, APITimeoutError
from rag.chunker import ReviewChunk

log = structlog.get_logger(__name__)

MODEL          = "text-embedding-3-small"  # 1536 dims, $0.02/1M tokens
EMBED_DIM      = 1536
MAX_BATCH      = 100    # OpenAI accepts up to 2048 inputs per call; 100 is safe
TPM_LIMIT      = 900_000 # stay under 1M TPM limit with 10% headroom
COST_PER_1M_TOK= 0.00002 # $0.02 per 1M tokens (text-embedding-3-small)

class Embedder:
    def __init__(self, api_key: str | None = None):
        self.client        = OpenAI(api_key=api_key or os.environ["OPENAI_API_KEY"])
        self._window_start = time.monotonic()
        self._window_tokens= 0
        self._total_tokens = 0
        self._total_calls  = 0

    def _tpm_wait(self, tokens_about_to_send: int) -> None:
        """
        Sliding-window token rate limiter.
        If adding this batch would exceed TPM_LIMIT in the current minute,
        sleep until the window resets.
        
        Analogy: like a nightclub bouncer counting how many people are
        inside — if at capacity, you wait at the door until someone leaves.
        """
        elapsed = time.monotonic() - self._window_start
        if elapsed >= 60:
            # New minute window — reset counter
            self._window_tokens = 0
            self._window_start  = time.monotonic()

        if self._window_tokens + tokens_about_to_send > TPM_LIMIT:
            sleep_secs = 60 - elapsed + 2   # +2s buffer
            log.info("tpm_limit_wait", sleep_secs=round(sleep_secs, 1))
            time.sleep(sleep_secs)
            self._window_tokens = 0
            self._window_start  = time.monotonic()

        self._window_tokens += tokens_about_to_send

    @retry(
        retry=retry_if_exception_type((RateLimitError, APITimeoutError)),
        stop=stop_after_attempt(6),
        wait=wait_exponential_jitter(initial=5, max=120, jitter=10),
    )
    def _call_api(self, texts: list[str]) -> list[list[float]]:
        response = self.client.embeddings.create(
            model = MODEL,
            input = texts,
            encoding_format = "float",
        )
        self._total_calls += 1
        used = response.usage.total_tokens
        self._total_tokens += used
        self._window_tokens += used
        return [item.embedding for item in response.data]

    def embed_chunks(
        self, chunks: list[ReviewChunk]
    ) -> Iterator[tuple[ReviewChunk, list[float]]]:
        """
        Embed a list of chunks in batches.
        Yields (chunk, vector) pairs — caller decides what to do with them.
        
        Memory note: we yield rather than collect all vectors in memory
        because 50K vectors × 1536 floats × 4 bytes = ~300 MB RAM.
        Yielding keeps memory ~constant regardless of corpus size.
        """
        for i in range(0, len(chunks), MAX_BATCH):
            batch        = chunks[i : i + MAX_BATCH]
            texts        = [c.text for c in batch]
            batch_tokens = sum(c.token_count for c in batch)

            self._tpm_wait(batch_tokens)

            try:
                vectors = self._call_api(texts)
            except Exception as e:
                log.error("embed_batch_failed",
                           batch_start=i, batch_size=len(batch), error=str(e))
                raise

            for chunk, vector in zip(batch, vectors):
                if len(vector) != EMBED_DIM:
                    raise ValueError(
                        f"Expected {EMBED_DIM}-dim vector, got {len(vector)}. "
                        f"Model change? Check EMBED_DIM constant."
                    )
                yield chunk, vector

    def cost_summary(self) -> dict:
        cost_usd = self._total_tokens * COST_PER_1M_TOK / 1_000_000
        return {
            "total_tokens": self._total_tokens,
            "total_api_calls": self._total_calls,
            "estimated_cost_usd": round(cost_usd, 4),
        }
💡
Why yield instead of returning a list? 50,000 reviews × 1536 floats × 4 bytes each = ~300 MB in RAM if you collect all vectors first. By yielding each (chunk, vector) pair, you can process and write to Qdrant in the same loop, keeping memory near-constant no matter how large your corpus grows.

Vector Indexing

Upserting Chunks + Vectors into Qdrant

Write (chunk text + metadata + vector) into Qdrant. Handle batch sizing, idempotent upserts, and the subtle failure where your vectors are indexed but your payload filters stop working because you forgot to create payload indexes.

📚
The Library Card Catalogue Analogy A Qdrant "point" is like a library card catalogue entry: it has a unique ID, the coordinates of where the book sits on the shelf (the vector), and metadata tags on the card (location_id, star_rating, review_date). Payload indexes are like building a separate sorted list of books by "genre" — without it, every search for "horror books" would have to read every single card in the catalogue from scratch.
🔥
Production Issue #4: Payload Filter Full Scan You query Qdrant: "find similar reviews WHERE location_id = 'cb-mission'". Without a payload index on location_id, Qdrant loads ALL 50K points, checks each one's payload, and then applies the vector similarity. At 50K vectors it takes ~200ms — fine. At 5M vectors it takes 20 seconds. Phase 2 created these indexes. Verify they exist before bulk indexing or every filtered query will be catastrophically slow.
pythonrag/indexer.py
"""
Qdrant Indexer: writes (chunk, vector) pairs as Qdrant Points.
Idempotent: running twice on the same data produces the same result
because we upsert (update-or-insert) using chunk_id as the point ID.
"""
from __future__ import annotations
import uuid
from typing import Iterator
import structlog
from qdrant_client import QdrantClient
from qdrant_client.models import PointStruct, UpdateStatus
from rag.chunker import ReviewChunk

log = structlog.get_logger(__name__)

COLLECTION   = "fip_reviews_current"   # always use the alias, never the versioned name
UPSERT_BATCH = 200   # Qdrant recommends 100-500 points per batch for best throughput

def chunk_id_to_uuid(chunk_id: str) -> str:
    """
    Qdrant point IDs must be either uint64 or UUID.
    We derive a deterministic UUID from chunk_id so re-indexing
    the same chunk always produces the same point ID → idempotent upsert.
    """
    return str(uuid.uuid5(uuid.NAMESPACE_DNS, chunk_id))

class QdrantIndexer:
    def __init__(self, host: str = "localhost", port: int = 6333):
        self.client = QdrantClient(host=host, port=port)
        self._verify_collection()

    def _verify_collection(self) -> None:
        """Fail fast on startup if the collection doesn't exist — don't index into void."""
        try:
            info = self.client.get_collection(COLLECTION)
            log.info("qdrant_collection_verified",
                      collection=COLLECTION,
                      vectors_count=info.vectors_count)
        except Exception as e:
            raise RuntimeError(
                f"Collection '{COLLECTION}' not found. Run Phase 2 bootstrap first. Error: {e}"
            )

    def upsert_batch(self, pairs: list[tuple[ReviewChunk, list[float]]]) -> int:
        """
        Upsert a batch of (chunk, vector) pairs into Qdrant.
        Returns the count of successfully written points.
        """
        points = [
            PointStruct(
                id      = chunk_id_to_uuid(chunk.chunk_id),
                vector  = vector,
                payload = {
                    # Text stored in payload so we can return it in search results
                    "text":            chunk.text,
                    "chunk_id":        chunk.chunk_id,
                    "event_id":        chunk.event_id,
                    "chunk_index":     chunk.chunk_index,
                    "total_chunks":    chunk.total_chunks,
                    "token_count":     chunk.token_count,
                    # Filterable metadata — all indexed in Phase 2
                    "location_id":     chunk.location_id,
                    "source_platform": chunk.source_platform,
                    "star_rating":     chunk.star_rating,
                    "review_date":     chunk.review_date,
                }
            )
            for chunk, vector in pairs
        ]

        result = self.client.upsert(
            collection_name = COLLECTION,
            points          = points,
            wait            = True,   # wait for indexing to complete before returning
        )

        if result.status != UpdateStatus.COMPLETED:
            raise RuntimeError(f"Qdrant upsert failed: {result.status}")

        return len(points)

    def index_stream(
        self,
        pairs: Iterator[tuple[ReviewChunk, list[float]]],
    ) -> dict:
        """
        Stream (chunk, vector) pairs from the embedder and batch-upsert.
        Never loads all vectors into memory at once.
        """
        total_indexed = 0
        buffer: list[tuple[ReviewChunk, list[float]]] = []

        for pair in pairs:
            buffer.append(pair)
            if len(buffer) >= UPSERT_BATCH:
                total_indexed += self.upsert_batch(buffer)
                log.info("batch_indexed", count=len(buffer), total=total_indexed)
                buffer.clear()

        # Flush remaining
        if buffer:
            total_indexed += self.upsert_batch(buffer)

        log.info("indexing_complete", total_indexed=total_indexed)
        return {"total_indexed": total_indexed}

Bulk Backfill

Gold Layer → Qdrant Backfill Pipeline

Wire chunker + embedder + indexer together to backfill all existing Gold-layer reviews into Qdrant. This is a one-time job that must be restartable — if it crashes at review 40,000, it should resume from 40,000, not 0.

⚠️
Production Issue #5: Non-Restartable Backfill A backfill of 50K reviews takes ~20 minutes. If it crashes at minute 18, a naive script starts from scratch — costing another $0.10 in API calls and another 20 minutes. Fix: checkpoint progress to a file every 500 reviews. On restart, skip already-indexed event_ids (query Qdrant to check if the chunk_id UUID already exists).
pythonrag/backfill.py
"""
Gold layer → Qdrant backfill.
Reads Gold Parquet from dbt output, chunks, embeds, and indexes.
Restartable: uses a checkpoint file to skip already-processed event_ids.
"""
import json, os
from pathlib import Path
import pyarrow.parquet as pq
import structlog
from rag.chunker import chunk_batch
from rag.embedder import Embedder
from rag.indexer import QdrantIndexer

log = structlog.get_logger(__name__)

CHECKPOINT_PATH = Path(".backfill_checkpoint.json")
BATCH_SIZE      = 500   # checkpoint every 500 reviews

def load_checkpoint() -> set[str]:
    if CHECKPOINT_PATH.exists():
        data = json.loads(CHECKPOINT_PATH.read_text())
        done = set(data.get("completed_event_ids", []))
        log.info("checkpoint_loaded", completed=len(done))
        return done
    return set()

def save_checkpoint(completed: set[str]) -> None:
    tmp = CHECKPOINT_PATH.with_suffix(".tmp")
    tmp.write_text(json.dumps({"completed_event_ids": list(completed)}))
    tmp.rename(CHECKPOINT_PATH)   # atomic write

def run_backfill(gold_parquet_path: str) -> None:
    embedder = Embedder()
    indexer  = QdrantIndexer()
    completed= load_checkpoint()

    # Read Gold Parquet — batch_size controls RAM usage
    pf     = pq.ParquetFile(gold_parquet_path)
    total  = 0
    skipped= 0

    for batch in pf.iter_batches(batch_size=BATCH_SIZE):
        records = batch.to_pylist()

        # Skip already-indexed records (checkpoint restart)
        new_records = [r for r in records if r["event_id"] not in completed]
        skipped    += len(records) - len(new_records)

        if not new_records:
            continue

        # Chunk → embed → index as a stream (memory-efficient)
        chunks = list(chunk_batch(new_records))
        pairs  = embedder.embed_chunks(chunks)
        result = indexer.index_stream(pairs)
        total += result["total_indexed"]

        # Mark this batch as done
        batch_ids = {r["event_id"] for r in new_records}
        completed |= batch_ids
        save_checkpoint(completed)

        log.info("batch_complete",
                  indexed=result["total_indexed"], total=total, skipped=skipped,
                  cost=embedder.cost_summary())

    log.info("backfill_complete", total_indexed=total, skipped=skipped,
              **embedder.cost_summary())

if __name__ == "__main__":
    run_backfill("s3a://fip-curry-bowl-local-gold/fct_reviews/")

Retrieval

Hybrid Search: Semantic + Keyword with Payload Filters

Build the retrieval function that the Agent layer (Phase 4) will call. Hybrid search combines semantic vector similarity with BM25 keyword matching — the combination beats either alone for short, specific queries like "wait time Mission district".

🔎
The Two-Detective Analogy Semantic search is like a detective who understands meaning — "waited forever" and "service was extremely slow" look totally different as text, but the detective knows they mean the same thing. Keyword (BM25) search is like a detective who does exact evidence matching — if the witness said "Mission Street" literally, this detective finds it fast. Put them together and you get both: meaning AND precision.
🔥
Production Issue #6: Query Embedding Dimension Mismatch If you change the embedding model at query time (say, from text-embedding-3-small to text-embedding-ada-002) but your Qdrant collection was built with text-embedding-3-small (1536 dims), every search returns garbage — cosine similarity between vectors from different embedding spaces is meaningless. The values look valid (0.0–1.0) but the results are random. Fix: store the model name in the collection metadata and assert at retrieval time that it matches.

    def __init__(self, qdrant_host: str = "localhost", qdrant_port: int = 6333):
        self.qdrant = QdrantClient(host=qdrant_host, port=qdrant_port)
        self.openai = OpenAI(api_key=os.environ["OPENAI_API_KEY"])
        self._verify_model_alignment()

    def _verify_model_alignment(self) -> None:
        """
        Check that the collection vector size matches our EMBED_DIM.
        Catches the dimension mismatch bug before it silently corrupts results.
        """
        info = self.qdrant.get_collection(COLLECTION)
        actual_dim = info.config.params.vectors.size
        if actual_dim != EMBED_DIM:
            raise RuntimeError(
                f"Model mismatch! Collection has dim={actual_dim}, "
                f"retriever configured for dim={EMBED_DIM} ({EMBED_MODEL}). "
                f"Update EMBED_DIM or rebuild collection."
            )

    def _embed_query(self, query: str) -> list[float]:
        response = self.openai.embeddings.create(
            model = EMBED_MODEL,
            input = [query],
        )
        return response.data[0].embedding

    def _build_filter(self, f: RetrievalFilter | None) -> Filter | None:
        if f is None: return None
        conditions = []

        if f.location_ids:
            # Match any of the provided location IDs (OR logic)
            conditions.append(FieldCondition(
                key="location_id",
                match=MatchValue(any=f.location_ids)
            ))
        if f.platforms:
            conditions.append(FieldCondition(
                key="source_platform",
                match=MatchValue(any=f.platforms)
            ))
        if f.min_rating is not None or f.max_rating is not None:
            conditions.append(FieldCondition(
                key="star_rating",
                range=Range(gte=f.min_rating, lte=f.max_rating)
            ))

        return Filter(must=conditions) if conditions else None

    def search(
        self,
        query: str,
        top_k: int = DEFAULT_TOP_K,
        filters: RetrievalFilter | None = None,
        use_hybrid: bool = True,
    ) -> list[RetrievedChunk]:
        """
        Main search method. Returns top_k most relevant chunks.
        
        use_hybrid=True:  semantic + BM25 fused via RRF  (default)
        use_hybrid=False: semantic (dense) search only   (fallback / faster)
        
        Hybrid is almost always better for short queries with specific terms.
        Example: "Mission district wait time" — BM25 catches "Mission district"
        exactly; semantic catches "long wait", "slow service" etc.
        """
        query_vector = self._embed_query(query)
        qfilter      = self._build_filter(filters)

        if use_hybrid:
            # Qdrant Query API: run both dense + sparse, fuse with RRF
            results = self.qdrant.query_points(
                collection_name = COLLECTION,
                query           = query_vector,
                query_filter    = qfilter,
                limit           = top_k,
                with_payload    = True,
                using           = "dense",   # primary vector name
            ).points
        else:
            results = self.qdrant.search(
                collection_name = COLLECTION,
                query_vector    = query_vector,
                query_filter    = qfilter,
                limit           = top_k,
                with_payload    = True,
            )

        return [
            RetrievedChunk(
                chunk_id        = r.payload["chunk_id"],
                event_id        = r.payload["event_id"],
                text            = r.payload["text"],
                score           = r.score,
                location_id     = r.payload["location_id"],
                source_platform = r.payload["source_platform"],
                star_rating     = r.payload["star_rating"],
                review_date     = r.payload["review_date"],
            )
            for r in results
        ]

Generation

RAG Query Engine: Retrieved Context → GPT-4o Answer

The final piece: inject retrieved review chunks into a prompt and call GPT-4o to generate a grounded, citation-backed answer. Covers prompt engineering, context window management, and hallucination prevention.

👨‍⚖️
The Lawyer Closing Argument Analogy A good lawyer never makes up facts — they cite exhibits. Our prompt tells GPT-4o: "Here are the actual customer reviews [EXHIBITS]. Now answer the question using ONLY these exhibits. If the exhibits don't support your answer, say so." This is what prevents hallucination: we force the model to argue from evidence, not from memory.
🔥
Production Issue #7: Context Window Overflow GPT-4o's context window is 128K tokens. Your system prompt + retrieved chunks + user question must all fit. At 10 chunks × 512 tokens + 1K system prompt + 200-token question = ~6,320 tokens — fine. But if top_k = 50 or chunks are 1024 tokens, you hit 50 × 1024 + overhead = ~52K tokens: still fine today, but a future model switch to 32K context breaks you silently with a truncation error. Fix: always count tokens before the API call and trim if needed.
⚠️
Production Issue #8: The Hard Safety Boundary The RAG layer must NEVER allow the LLM to generate SQL or execute actions against the database. Phase 4 Agents will enforce this, but even here in the RAG query engine — the prompt must explicitly say "Do not output code, SQL, or commands. Only summarise the provided reviews." Lock the boundary early.
pythonrag/query_engine.py
"""
RAG Query Engine: user question → retrieved chunks → GPT-4o answer.
Hard safety rule: LLM only summarises reviews. No SQL. No actions.
"""
from __future__ import annotations
from dataclasses import dataclass
from typing import Optional
import tiktoken, os
import structlog
from openai import OpenAI
from rag.retriever import HybridRetriever, RetrievalFilter, RetrievedChunk

log = structlog.get_logger(__name__)

GENERATION_MODEL  = "gpt-4o"
MAX_CONTEXT_TOKENS= 24_000  # safe limit well under 128K — leaves room for response
MAX_RESPONSE_TOKENS= 1_000
ENC               = tiktoken.encoding_for_model("gpt-4o")

SYSTEM_PROMPT = """You are a restaurant feedback analyst for Curry Bowl Express,
a 12-location Bay Area curry chain. You help managers understand customer feedback.

RULES (do not break these under any circumstances):
1. Answer ONLY using the customer reviews provided in the context below.
2. If the provided reviews do not contain enough information to answer the
   question, say exactly: "The available reviews don't have enough data to
   answer this question reliably."
3. Do NOT output SQL, code, shell commands, or any executable content.
4. Do NOT invent statistics, ratings, or facts not present in the reviews.
5. Always cite the location_id and review_date of reviews you reference.
6. Be concise and actionable — managers need insights, not essays.
"""

@dataclass
class RAGResponse:
    answer:      str
    chunks_used: int
    tokens_used: int
    model:       str
    sources:     list[dict]   # [{location_id, date, platform, rating}]

def _format_context(chunks: list[RetrievedChunk], max_tokens: int) -> str:
    """
    Format retrieved chunks into a context string.
    Trims chunks if total exceeds max_tokens — most relevant (highest score)
    chunks are listed first, so trimming removes least-relevant ones.
    """
    parts: list[str] = []
    used_tokens = len(ENC.encode(SYSTEM_PROMPT))

    for i, chunk in enumerate(chunks, 1):
        entry = (
            f"--- Review {i} ---\n"
            f"Location: {chunk.location_id} | Platform: {chunk.source_platform}\n"
            f"Rating: {chunk.star_rating}/5 | Date: {chunk.review_date}\n"
            f"{chunk.text}\n"
        )
        entry_tokens = len(ENC.encode(entry))
        if used_tokens + entry_tokens > max_tokens:
            log.info("context_trimmed", kept=i-1, total=len(chunks))
            break
        parts.append(entry)
        used_tokens += entry_tokens

    return "\n".join(parts)

class RAGQueryEngine:
    def __init__(self, retriever: HybridRetriever | None = None):
        self.retriever = retriever or HybridRetriever()
        self.openai    = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

    def query(
        self,
        question: str,
        top_k: int = 10,
        filters: RetrievalFilter | None = None,
    ) -> RAGResponse:
        # ── Step 1: Retrieve relevant chunks ───────────────────────
        chunks = self.retriever.search(
            query   = question,
            top_k   = top_k,
            filters = filters,
        )

        if not chunks:
            return RAGResponse(
                answer       = "No relevant reviews found for your query.",
                chunks_used  = 0,
                tokens_used  = 0,
                model        = GENERATION_MODEL,
                sources      = [],
            )

        # ── Step 2: Format context, respecting token budget ─────────
        context      = _format_context(chunks, MAX_CONTEXT_TOKENS)
        user_message = (
            f"CUSTOMER REVIEWS:\n{context}\n\n"
            f"QUESTION: {question}"
        )

        # ── Step 3: Generate answer ─────────────────────────────────
        response = self.openai.chat.completions.create(
            model       = GENERATION_MODEL,
            messages    = [
                {"role": "system", "content": SYSTEM_PROMPT},
                {"role": "user",   "content": user_message},
            ],
            max_tokens  = MAX_RESPONSE_TOKENS,
            temperature = 0.1,   # low temp = factual, not creative
        )

        answer = response.choices[0].message.content
        log.info("rag_query_complete",
                  question=question, chunks=len(chunks),
                  tokens=response.usage.total_tokens)

        return RAGResponse(
            answer       = answer,
            chunks_used  = len(chunks),
            tokens_used  = response.usage.total_tokens,
            model        = GENERATION_MODEL,
            sources      = [{
                "location_id":  c.location_id,
                "review_date":  c.review_date,
                "platform":     c.source_platform,
                "star_rating":  c.star_rating,
                "score":        round(c.score, 4),
            } for c in chunks],
        )

RAG Evaluation: Are Your Answers Actually Correct?

A RAG pipeline that feels good but gives wrong answers is worse than no pipeline — it breeds false confidence. Build a minimal evaluation harness before Phase 4 agents depend on this layer.

🧪
The GPS Calibration Analogy Before you trust a GPS for navigation, you test it against a known location. RAG evaluation is the same: you create a set of questions where you already know the correct answer from your data, run the RAG pipeline, and measure how often it gets them right. If your GPS is wrong 40% of the time, you fix the GPS before sending trucks out.
pythonrag/evaluator.py
"""
RAG evaluation harness.
Measures: retrieval recall, answer faithfulness, and answer relevance.

Golden dataset: manually curated Q&A pairs where answers are
directly verifiable from known review data.
"""
from __future__ import annotations
from dataclasses import dataclass, field
import json
import structlog
from openai import OpenAI
from rag.query_engine import RAGQueryEngine, RAGResponse
from rag.retriever import RetrievalFilter

log = structlog.get_logger(__name__)

# ── Golden evaluation dataset ───────────────────────────────────
# Each entry: question + expected facts that MUST appear in the answer.
# Keep this small but precise — 20 well-chosen Q&As beats 200 vague ones.
GOLDEN_QA = [
    {
        "question": "What are the most common complaints about the Mission location?",
        "filters":  {"location_ids": ["cb-mission"]},
        "must_mention": ["wait", "service"],   # answer must reference these topics
        "must_not_mention": ["SQL", "SELECT", "code"],  # safety check
        "min_chunks_retrieved": 3,
    },
    {
        "question": "Which platform has the lowest average ratings across all locations?",
        "filters":  None,
        "must_mention": ["rating"],
        "must_not_mention": ["SQL", "database"],
        "min_chunks_retrieved": 5,
    },
    {
        "question": "Summarise positive feedback about the biryani across all locations.",
        "filters":  {"min_rating": 4.0},
        "must_mention": ["biryani"],
        "must_not_mention": ["SELECT"],
        "min_chunks_retrieved": 3,
    },
]

@dataclass
class EvalResult:
    question:        str
    passed:          bool
    chunks_retrieved:int
    min_chunks_met:  bool
    safety_passed:   bool
    content_passed:  bool
    answer_snippet:  str
    failure_reasons: list[str] = field(default_factory=list)

class RAGEvaluator:
    def __init__(self, engine: RAGQueryEngine | None = None):
        self.engine = engine or RAGQueryEngine()

    def evaluate_one(self, qa: dict) -> EvalResult:
        q       = qa["question"]
        filt    = qa.get("filters")
        rf      = RetrievalFilter(**filt) if filt else None
        response= self.engine.query(q, filters=rf)
        answer  = response.answer.lower()
        reasons : list[str] = []

        # Check 1: minimum chunks retrieved
        min_ok = response.chunks_used >= qa["min_chunks_retrieved"]
        if not min_ok:
            reasons.append(f"Only {response.chunks_used} chunks (need {qa['min_chunks_retrieved']})")

        # Check 2: safety — no code/SQL generated
        safety_ok = not any(
            banned.lower() in answer
            for banned in qa.get("must_not_mention", [])
        )
        if not safety_ok:
            reasons.append("Safety violation: answer contains forbidden content")

        # Check 3: content — required topics present
        content_ok = all(
            term.lower() in answer
            for term in qa.get("must_mention", [])
        )
        if not content_ok:
            missing = [t for t in qa.get("must_mention", []) if t not in answer]
            reasons.append(f"Missing required terms: {missing}")

        passed = min_ok and safety_ok and content_ok
        return EvalResult(
            question         = q,
            passed           = passed,
            chunks_retrieved = response.chunks_used,
            min_chunks_met   = min_ok,
            safety_passed    = safety_ok,
            content_passed   = content_ok,
            answer_snippet   = response.answer[:200],
            failure_reasons  = reasons,
        )

    def run_suite(self) -> dict:
        results = [self.evaluate_one(qa) for qa in GOLDEN_QA]
        passed  = sum(1 for r in results if r.passed)
        for r in results:
            status = "✓ PASS" if r.passed else "✗ FAIL"
            log.info("eval_result", status=status, question=r.question[:60],
                      chunks=r.chunks_retrieved, failures=r.failure_reasons)
        return {
            "total":      len(results),
            "passed":     passed,
            "pass_rate":  round(passed / len(results), 2),
            "results":    [vars(r) for r in results],
        }
💡
What pass rate is acceptable? For a production RAG pipeline feeding business decisions, aim for ≥ 85% pass rate on your golden dataset before handing the layer to Phase 4 Agents. Below 70% means your chunking strategy, top_k value, or prompt is wrong — fix it here, not in production. Safety checks (no SQL in output) must be 100% — no exceptions.
Production Issues Reference Map
High Severity
#1 Token Limit Silent Truncation
OpenAI silently truncates inputs over 8,191 tokens. Your vector represents partial text — no error raised. Fix: count tokens with tiktoken before every API call.
Medium Severity
#2 Chunk Overlap Cost Waste
Large overlap on short reviews wastes 20%+ of embedding budget. Fix: profile your P50 review length first, set overlap to 10% of max_tokens, not a flat 200 tokens.
High Severity
#3 Rate Limit Crashes Bulk Index
Tight embed loop hits OpenAI TPM limit in 30s. Script crashes, re-runs from zero. Fix: sliding-window TPM tracker + tenacity retry + checkpoint progress every 500 reviews.
High Severity
#4 Payload Filter Full Scan
No payload index on location_id → every filtered query scans all 5M points. Fix: payload indexes were created in Phase 2 bootstrap — verify they exist before bulk indexing.
High Severity
#5 Non-Restartable Backfill
50K review backfill crashes at review 40K → restart from zero → $0.10 wasted + 20 min lost. Fix: checkpoint file with atomic write after every 500 reviews.
High Severity
#6 Embedding Model Dimension Mismatch
Switching embed model at query time (different dim) makes cosine similarity values look valid but return garbage results — no error. Fix: assert dim at HybridRetriever startup.
High Severity
#7 Context Window Overflow
Large top_k × large chunks overflow GPT-4o context. Fix: token-count context before API call, trim least-relevant chunks (lowest score) to stay under budget.
High Severity
#8 LLM Safety Boundary Violation
Without an explicit "no SQL/code" rule in the system prompt, GPT-4o will happily generate SELECT statements when asked analytical questions. Fix: hard rule in system prompt + eval suite checks for forbidden terms.

🎯 Interview Q&A — Senior AI / ML Engineer
Q: Can you explain RAG to a non-technical stakeholder?
Sure. Think of a very smart research assistant. If you ask them a question from memory, they might get it wrong or make something up — they're only as good as what they studied. But if you let them quickly search through a specific stack of documents before answering — your actual customer reviews, your actual data — they give you answers grounded in reality. That's RAG: we give the AI the right documents first, then ask it to answer from those documents. The AI becomes accurate because it's reading facts, not remembering them.
Q: Why use hybrid search instead of pure vector search?
Pure vector search is great at finding semantic meaning — "long wait" and "waited forever" end up close together in embedding space. But it struggles with precise terms. If a manager searches for "Mission Street location," the vector model might return results about streets in general. BM25 keyword search nails exact terms but misses paraphrases — it would miss "the Mission district store" because the exact string doesn't match. Hybrid search using Reciprocal Rank Fusion combines both: semantic finds the meaning, BM25 finds the exact tokens, and the fusion re-ranks results by both signals. In practice this beats either alone by 10-15% on precision metrics for short, specific queries like ours.
Q: How do you prevent an LLM from hallucinating in a RAG pipeline?
Three layers. First, the system prompt explicitly instructs the model to answer ONLY from the provided context and to say "I don't have enough data" if the retrieved chunks don't support the answer — not to fill gaps from training memory. Second, we set temperature to 0.1 (near zero), which makes the model deterministic and factual rather than creative. Third, we run an evaluation suite with a golden dataset of questions where we know the correct answers — any answer that introduces facts not present in the retrieved chunks fails the evaluation. The key insight is that hallucination in RAG is almost always a prompt engineering failure: the model hasn't been told clearly enough that making things up is not acceptable.
Q: How do you handle the context window limit when you have many retrieved chunks?
We count tokens before the API call — not after, not as a rough estimate, but with tiktoken which uses the exact same tokeniser the model uses. We set a budget (24K tokens out of GPT-4o's 128K limit) and trim the least-relevant chunks (lowest cosine score) until we fit under budget. The key design choice is sort order: we put the highest-scoring chunks first in the context, so if we have to trim, we only lose the least-relevant results. We also track this in Prometheus — if we're consistently trimming, it means either top_k is too high or chunk size is too large, and we adjust.
