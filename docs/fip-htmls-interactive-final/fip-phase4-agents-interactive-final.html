LangGraph Explained From First Principles
Why LangGraph — and what a stateful agent graph actually means for a fresher
🚦
The Air Traffic Control Tower Analogy A bare LangChain chain is a conveyor belt: every customer question goes through exactly the same steps in the same order. LangGraph is an air traffic control tower: each incoming flight (question) is routed to a different runway based on what it needs. A ratings question goes to the analytics runway. A customer-complaint question goes to the RAG retrieval runway. An emergency alert question gets priority handling. All flights land at the same terminal (synthesize node) but the route adapts to each flight. That adaptability is what "stateful multi-agent orchestration" means in practice.
🚪
intent_classifier
ENTRY NODE
spaCy NER extracts locations + dates. GPT-4o few-shot classifies intent: SUMMARY | TREND | COMPARE | ALERT. Routes to correct tool node via conditional edge.
🔍
rag_retrieval
TOOL NODE
Calls Phase 3 RAGQueryEngine with extracted location filters. Returns grounded answer from Qdrant fip_reviews_current alias. SUMMARY intent routes here.
📊
analytics_query
TOOL NODE
DuckDB read-only on dbt Gold Parquet from LocalStack S3. Computes TREND and COMPARE aggregations. Cannot write — Phase 0 safety boundary enforced at driver level, not just prompt.
⚠
alert_detector
CONDITIONAL
Checks if avg_rating dropped >0.5 stars vs 7-day baseline for any location. Produces alert_details list. ALERT intent routes here. Result feeds into synthesize.
✓
synthesize
END NODE
All paths converge here. Merges RAG answer + analytics rows + alert details into a structured 300-word manager briefing. Cites location_ids. Only node that calls GPT-4o for generation.
Data flow: User question → intent_classifier → [conditional route] → rag_retrieval | analytics_query | alert_detector → synthesize → Structured manager answer with citations

Data Generator — Agent Test Scenarios
🧪 FIPAgentGenerator — Realistic Manager Questions with Ground-Truth Intent Labels
🎭
The Fire Drill Script Analogy A fire drill needs a pre-written script: who pulls the alarm, who exits which door, who checks the roll-call. The scenario generator writes the agent fire drill script: realistic manager questions with the expected intent and expected tool node already known. Run the agent against this script before every deploy. If intent classification mislabels a question or the wrong tool fires, the test catches it before a real manager sees a wrong answer about a real rating drop at a real location.
pythonscripts/generate_agent_scenarios.py
import argparse, json, random, uuid

LOCATIONS = [
    "cb-mission","cb-castro","cb-soma","cb-tenderloin",
    "cb-richmond","cb-sunset","cb-oakland","cb-berkeley",
    "cb-sj-downtown","cb-sj-willow","cb-sunnyvale","cb-fremont",
]

TEMPLATES = [
    # SUMMARY -- text retrieval via Phase 3 RAGQueryEngine
    {"t":"Summarise feedback for {loc} this month.", "intent":"SUMMARY", "tool":"rag_retrieval"},
    {"t":"What are customers saying about {loc}?",     "intent":"SUMMARY", "tool":"rag_retrieval"},
    {"t":"Most common complaints at {loc} on DoorDash?", "intent":"SUMMARY", "tool":"rag_retrieval"},
    # TREND -- time-series via DuckDB Gold (read-only)
    {"t":"Has {loc} rating improved over 30 days?",     "intent":"TREND",   "tool":"analytics_query"},
    {"t":"Show me the weekly rating trend for {loc}.",  "intent":"TREND",   "tool":"analytics_query"},
    # COMPARE -- cross-location analytics
    {"t":"Which location has the lowest rating this week?","intent":"COMPARE","tool":"analytics_query"},
    {"t":"Compare {loc} vs {loc2} ratings this month.",  "intent":"COMPARE","tool":"analytics_query"},
    # ALERT -- threshold monitoring via alert_detector
    {"t":"Any location dropping below 3.5 stars?",      "intent":"ALERT",   "tool":"alert_detector"},
    {"t":"Which locations need urgent attention now?",   "intent":"ALERT",   "tool":"alert_detector"},
    {"t":"Any sudden rating drops in the past 7 days?",  "intent":"ALERT",   "tool":"alert_detector"},
]

class FIPAgentGenerator:
    def __init__(self, seed:int=42): random.seed(seed)

    def generate(self, n:int) -> list:
        out = []
        for _ in range(n):
            tmpl = random.choice(TEMPLATES)
            loc  = random.choice(LOCATIONS)
            loc2 = random.choice([l for l in LOCATIONS if l != loc])
            out.append({
                "scenario_id":     str(uuid.uuid4())[:8],
                "question":        tmpl["t"].format(loc=loc, loc2=loc2),
                "expected_intent": tmpl["intent"],
                "expected_tool":   tmpl["tool"],
                "location_hint":   loc,
            })
        return out

if __name__ == "__main__":
    ap = argparse.ArgumentParser()
    ap.add_argument("--count", type=int, default=20)
    ap.add_argument("--seed",  type=int, default=42)
    args = ap.parse_args()
    for s in FIPAgentGenerator(args.seed).generate(args.count):
        print(json.dumps(s, indent=2))

01
Environment & State Schema

Install LangGraph, Define AgentState TypedDict

LangGraph passes a single typed state object between every node. Define it up front with all fields typed. Every node reads from and writes partial updates back to this schema. Getting the types wrong produces silent None values inside a running graph that are very hard to debug.

🏭
The Hospital Patient Chart Analogy A patient's chart travels with them through triage, X-ray, surgery, and recovery. Each department reads what previous departments wrote and appends only their own notes — the chart is never replaced, only accumulated. LangGraph AgentState is the chart: the question field is set once at intake and never changes; the final_answer field starts as None and gets filled by the synthesize node; the nodes_visited list grows as the graph executes.

pythonagent/state.py
from typing import TypedDict, Optional, Annotated
from langgraph.graph.message import add_messages

class AgentState(TypedDict):
    # ── Input (set once at invocation, never mutated) ────────────
    session_id:       str
    question:         str
    location_filter:  Optional[str]

    # ── Set by intent_classifier node ─────────────────────────────
    intent:           Optional[str]   # SUMMARY | TREND | COMPARE | ALERT
    entities:         Optional[dict]  # {locations:[], date_keywords:[]}

    # ── Set by tool nodes (accumulated, never overwritten) ────────
    rag_answer:       Optional[str]
    retrieved_chunks: Optional[list]
    analytics_rows:   Optional[list]
    alert_triggered:  Optional[bool]
    alert_details:    Optional[list]

    # ── Set by synthesize node (final output) ─────────────────────
    final_answer:     Optional[str]
    sources:          Optional[list]
    nodes_visited:    Optional[list]  # full audit trail of graph path
    error:            Optional[str]

    # ── Multi-turn: append-only via LangGraph add_messages reducer ─
    messages: Annotated[list, add_messages]
💡
Why TypedDict not Pydantic? LangGraph uses TypedDict natively — it handles the state merge logic internally. Using Pydantic BaseModel adds validation overhead on every node return and requires extra boilerplate. Keep state as TypedDict; use Pydantic at the API boundary (lambda_handler input).
02
Intent Classifier Node

spaCy NER + GPT-4o Few-Shot Intent Classification

The graph entry node. spaCy (Phase 0 locked NLP) extracts location entities locally for free. GPT-4o few-shot classifies the intent with max_tokens=5 — the cheapest possible classification call. spaCy handles entity extraction; GPT-4o handles the decision. Optimal cost split.

🏥
The Hospital Triage Analogy A triage nurse sees every patient first, takes their vitals (spaCy NER: locations, dates), and decides the care pathway before any specialist time is spent. A question about "ratings over time" goes straight to the diagnostics lab (analytics_query). A question about "what customers are saying" goes to the consulting room (rag_retrieval). An emergency question goes to the emergency bay (alert_detector). No specialist time is wasted on wrong-routed patients.
🔥
Production Issue #1: Intent Accuracy Below 90% Silently Misleads Managers An ALERT question classified as SUMMARY routes to rag_retrieval. The answer looks plausible (sentiment summary for all locations) but the alert_detector never fires. A manager misses a real 0.8-star rating drop at one location. Fix: target ≥90% accuracy on generated eval dataset before deploy; add 3+ ALERT examples for "urgent", "attention", "immediate" language in few-shot prompt.
pythonagent/nodes/intent_classifier.py
import os
import spacy            # Phase 0 locked: local NLP for entity extraction
from openai import OpenAI
from agent.state import AgentState
import structlog

log     = structlog.get_logger(__name__)
NLP     = spacy.load("en_core_web_sm")      # free, local, 12 MB model
CLIENT  = OpenAI(api_key=os.environ["OPENAI_API_KEY"])
INTENTS = ["SUMMARY", "TREND", "COMPARE", "ALERT"]

# Location aliases managers actually use --> canonical IDs
LOC_MAP = {
    "mission":"cb-mission", "castro":"cb-castro", "soma":"cb-soma",
    "tenderloin":"cb-tenderloin", "richmond":"cb-richmond",
    "sunset":"cb-sunset", "oakland":"cb-oakland", "berkeley":"cb-berkeley",
    "fremont":"cb-fremont", "sunnyvale":"cb-sunnyvale",
    "san jose":"cb-sj-downtown", "willow glen":"cb-sj-willow",
}

FEW_SHOT = """Classify the restaurant manager question into ONE of:
SUMMARY, TREND, COMPARE, ALERT.

SUMMARY = asking for review text summaries or customer sentiment
TREND   = asking about ratings changing over time (single location)
COMPARE = comparing two or more locations against each other
ALERT   = urgent issues, threshold breaches, sudden drops

Examples:
Q: What are customers saying about our biryani at Mission? -> SUMMARY
Q: Has Castro improved its rating this month?              -> TREND
Q: Which location has the worst rating this week?          -> COMPARE
Q: Are any locations below 3 stars?                        -> ALERT
Q: Which locations need urgent attention?                  -> ALERT
Q: Any sudden rating drops this week?                      -> ALERT

Question: {question}
Intent (one word only):"""

def extract_entities(question: str) -> dict:
    doc  = NLP(question.lower())
    locs = []
    for chunk in doc.noun_chunks:
        for alias, canon in LOC_MAP.items():
            if alias in chunk.text and canon not in locs:
                locs.append(canon)
    date_kw = [w for w in
               ["today","this week","this month","last week","30 days","7 days"]
               if w in question.lower()]
    return {"locations": locs, "date_keywords": date_kw}

def intent_classifier_node(state: AgentState) -> dict:
    q        = state["question"]
    entities = extract_entities(q)
    resp = CLIENT.chat.completions.create(
        model      = os.getenv("OPENAI_MODEL", "gpt-4o"),
        messages   = [{"role":"user", "content": FEW_SHOT.format(question=q)}],
        max_tokens = 5,   # cheapest call in the entire graph
        temperature= 0,
    )
    raw    = resp.choices[0].message.content.strip().upper()
    intent = raw if raw in INTENTS else "SUMMARY"  # safe default
    log.info("intent_classified", intent=intent, entities=entities, q=q[:60])
    return {
        "intent":        intent,
        "entities":      entities,
        "nodes_visited": (state.get("nodes_visited") or []) + ["intent_classifier"],
    }
03
Tool Nodes

RAG Tool + Analytics Tool (DuckDB Read-Only) + Alert Detector

Three tool nodes, each enforcing the Phase 0 hard safety boundary differently. RAG calls Phase 3 RAGQueryEngine via the Qdrant alias (never a versioned name). Analytics uses DuckDB in-memory on Gold Parquet — physically cannot write. Alert detector queries 14-day Gold snapshots and fires on 0.5-star drops.

📚
The Read-Only Library Card Analogy A library card lets you read any book, but you cannot write in or tear pages from the books. DuckDB with in-memory mode pointing at Parquet files is the library card: the agent can SELECT any data and compute any aggregation, but the underlying files are physically unchanged. Even if the LLM generated a DROP TABLE statement, DuckDB rejects the write at the driver level — not just at the prompt level. Prompt-only safety is necessary but not sufficient.
🔥
Production Issue #2: RAG Tool Hardcodes Collection Version Using "fip_reviews_v1" instead of the alias "fip_reviews_current" means the agent silently queries stale data after Phase 3 rolls a new collection. The alias was bootstrapped in Phase 2 for exactly this reason. Always use the alias. Never hardcode a versioned collection name in agent code.
🔥
Production Issue #3: DuckDB Write Access Bypasses Safety Boundary Default DuckDB opens in read-write mode. One unvalidated .execute() with a mutation query modifies the Gold Parquet files permanently. Fix: always use duckdb.connect(":memory:") for analytics — in-memory DuckDB reads from S3 Parquet but has no file-write capability whatsoever.
pythonagent/nodes/tool_nodes.py
import os, duckdb, structlog
from agent.state import AgentState
from rag.query_engine import RAGQueryEngine   # Phase 3 engine
from rag.retriever  import RetrievalFilter

log         = structlog.get_logger(__name__)
_rag_engine = None

def get_rag() -> RAGQueryEngine:
    global _rag_engine
    # Module-level singleton -- one connection, reused across invocations
    if _rag_engine is None:
        _rag_engine = RAGQueryEngine()  # connects to FLOCI Qdrant at localhost:6333
    return _rag_engine

def gold_path() -> str:
    return os.getenv("GOLD_PARQUET_PATH",
                     "s3a://fip-curry-bowl-local-gold/fct_location_ratings/")

# ── RAG Retrieval Node ──────────────────────────────────────────
def rag_retrieval_node(state: AgentState) -> dict:
    entities = state.get("entities") or {}
    locs     = entities.get("locations") or []
    filt     = RetrievalFilter(location_ids=locs) if locs else None
    resp     = get_rag().query(state["question"], top_k=10, filters=filt)
    log.info("rag_tool_called", chunks=resp.chunks_used, tokens=resp.tokens_used)
    return {
        "rag_answer":      resp.answer,
        "retrieved_chunks": resp.sources,
        "nodes_visited":   (state.get("nodes_visited") or []) + ["rag_retrieval"],
    }

# ── Analytics Query Node (DuckDB read-only in-memory) ───────────
def analytics_query_node(state: AgentState) -> dict:
    con      = duckdb.connect(":memory:")  # in-memory: cannot write files
    con.execute("INSTALL httpfs; LOAD httpfs;")
    # Configure S3A for FLOCI LocalStack (same as Phase 2 PySpark config)
    con.execute(f"""SET s3_endpoint='{os.getenv("DUCKDB_S3_ENDPOINT","s3.amazonaws.com")}' ;
                    SET s3_access_key_id='{os.getenv("DUCKDB_S3_ACCESS_KEY","")}' ;
                    SET s3_secret_access_key='{os.getenv("DUCKDB_S3_SECRET_KEY","")}' ;""")
    intent   = state.get("intent", "COMPARE")
    entities = state.get("entities") or {}
    locs     = entities.get("locations") or []

    if intent == "TREND" and locs:
        sql = (f"SELECT review_date, avg_rating, review_count "\span>
               f"FROM read_parquet('{gold_path()}**/*.parquet') "\span>
               f"WHERE location_id='{locs[0]}' ORDER BY review_date DESC LIMIT 30")
    else:  # COMPARE: all locations sorted by worst rating
        sql = (f"SELECT location_id, ROUND(AVG(avg_rating),2) avg_rating_7d, "\span>
               f"SUM(review_count) total_reviews "\span>
               f"FROM read_parquet('{gold_path()}**/*.parquet') "\span>
               f"WHERE review_date >= CURRENT_DATE - INTERVAL '7 days' "\span>
               f"GROUP BY location_id ORDER BY avg_rating_7d ASC LIMIT 12")

    rel  = con.execute(sql)
    cols = [d[0] for d in rel.description]
    data = [dict(zip(cols, r)) for r in rel.fetchall()]
    log.info("analytics_done", rows=len(data), intent=intent)
    return {
        "analytics_rows": data,
        "nodes_visited":  (state.get("nodes_visited") or []) + ["analytics_query"],
    }

# ── Alert Detector Node ──────────────────────────────────────────
ALERT_THRESHOLD = 0.5  # 0.5 star drop in 7 days triggers an alert

def alert_detector_node(state: AgentState) -> dict:
    con = duckdb.connect(":memory:")
    con.execute("INSTALL httpfs; LOAD httpfs;")
    # Compare this week vs prior week per location using dbt Gold snapshots
    sql = f"""WITH cur AS (
      SELECT location_id, AVG(avg_rating) w FROM read_parquet('{gold_path()}**/*.parquet')
      WHERE review_date >= CURRENT_DATE - INTERVAL '7 days' GROUP BY location_id),
    prv AS (
      SELECT location_id, AVG(avg_rating) w FROM read_parquet('{gold_path()}**/*.parquet')
      WHERE review_date BETWEEN CURRENT_DATE-INTERVAL '14 days'
                            AND CURRENT_DATE-INTERVAL '7 days'  GROUP BY location_id)
    SELECT c.location_id,
           ROUND(c.w,2) current_avg, ROUND(p.w,2) prev_avg,
           ROUND(p.w - c.w, 2) drop_amount
    FROM cur c JOIN prv p USING(location_id)
    WHERE p.w - c.w >= {ALERT_THRESHOLD}
    ORDER BY drop_amount DESC"""

    rel    = con.execute(sql)
    cols   = [d[0] for d in rel.description]
    alerts = [dict(zip(cols, r)) for r in rel.fetchall()]
    return {
        "alert_triggered": bool(alerts),
        "alert_details":   alerts,
        "nodes_visited":   (state.get("nodes_visited") or []) + ["alert_detector"],
    }
04
Synthesize Node + Graph Assembly

Merge All Tool Results → Structured Manager Briefing + Wire the Graph

Synthesize is the only node that calls GPT-4o for generation. It merges RAG answer, analytics rows, and alert details into a 300-word structured briefing with cited location_ids. Then wire all 5 nodes into the LangGraph StateGraph with conditional edges from intent_classifier.

📝
The Editor-in-Chief Analogy Individual journalists (tool nodes) file their reports: the RAG reporter brings customer quotes, the analytics reporter brings numbers, the alert reporter flags breaking news. The editor-in-chief (synthesize node) writes the final front-page story: leads with the most important finding, cites all sources, and writes one coherent narrative. The editor never goes out to investigate — they only synthesise what reporters brought back.
🔥
Production Issue #4: Graph Init Inside Lambda Handler Building the LangGraph StateGraph inside lambda_handler() adds 5–10 seconds of initialisation (spaCy model load, Qdrant connection, LangGraph compile) to every single invocation. Lambda containers stay warm between calls; module-level code runs once per container lifetime. Fix: _GRAPH = build_graph() at module level, before lambda_handler.
pythonagent/nodes/synthesize.py
import json, os
from openai import OpenAI
from agent.state import AgentState

CLIENT = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

SYSTEM = """You are a concise restaurant operations analyst for Curry Bowl Express.
RULES (never break these):
1. Lead with the most actionable finding.
2. Cite specific location_ids and review_dates from the data provided.
3. If alert data is present, open with the alert paragraph.
4. Maximum 300 words. Plain English -- no jargon.
5. End with 1-2 specific recommended actions for the manager.
6. NEVER output SQL, code, shell commands, or invented statistics."""

def synthesize_node(state: AgentState) -> dict:
    parts = [f"Manager Question: {state['question']}"]

    if state.get("alert_details"):
        parts.append("\n*** RATING ALERTS DETECTED ***\n"
                     + json.dumps(state["alert_details"], indent=2))

    if state.get("rag_answer"):
        parts.append(f"\nCustomer Review Analysis:\n{state['rag_answer']}")

    if state.get("analytics_rows"):
        parts.append("\nAnalytics Data (dbt Gold layer):\n"
                     + json.dumps(state["analytics_rows"][:12], indent=2))

    resp = CLIENT.chat.completions.create(
        model       = os.getenv("OPENAI_MODEL", "gpt-4o"),
        messages    = [
            {"role": "system", "content": SYSTEM},
            {"role": "user",   "content": "\n".join(parts)},
        ],
        max_tokens  = 500,
        temperature = 0.2,  # slightly creative for readability, not 0 like intent node
    )
    return {
        "final_answer":   resp.choices[0].message.content,
        "sources":        state.get("retrieved_chunks", []),
        "nodes_visited":  (state.get("nodes_visited") or []) + ["synthesize"],
    }
pythonagent/graph.py
from langgraph.graph import StateGraph, END
from agent.state import AgentState
from agent.nodes.intent_classifier import intent_classifier_node
from agent.nodes.tool_nodes        import (rag_retrieval_node,
                                             analytics_query_node,
                                             alert_detector_node)
from agent.nodes.synthesize         import synthesize_node

def route_by_intent(state: AgentState) -> str:
    # Conditional edge function -- LangGraph calls this after intent_classifier
    # Returns the NAME of the next node to execute
    routing = {
        "SUMMARY": "rag_retrieval",
        "TREND":   "analytics_query",
        "COMPARE": "analytics_query",
        "ALERT":   "alert_detector",
    }
    return routing.get(state.get("intent", "SUMMARY"), "rag_retrieval")

def build_graph():
    g = StateGraph(AgentState)
    # Register all 5 nodes
    g.add_node("intent_classifier", intent_classifier_node)
    g.add_node("rag_retrieval",    rag_retrieval_node)
    g.add_node("analytics_query",  analytics_query_node)
    g.add_node("alert_detector",   alert_detector_node)
    g.add_node("synthesize",        synthesize_node)

    g.set_entry_point("intent_classifier")

    # Conditional routing from intent_classifier
    g.add_conditional_edges(
        "intent_classifier", route_by_intent,
        {"rag_retrieval":   "rag_retrieval",
         "analytics_query": "analytics_query",
         "alert_detector":  "alert_detector"}
    )
    # All tool paths converge to synthesize
    g.add_edge("rag_retrieval",   "synthesize")
    g.add_edge("analytics_query", "synthesize")
    g.add_edge("alert_detector",  "synthesize")
    g.add_edge("synthesize",       END)

    return g.compile(recursion_limit=10)  # prevents runaway loops

Lambda Handler + FLOCI Invoke

AWS Lambda Handler — Same Code, FLOCI Local vs Production AWS

Wrap the compiled graph in a Lambda handler. Graph initialised at module level once (warm start benefit). Same code runs on FLOCI locally via python -c and in AWS Lambda via API Gateway — the only difference is environment variables.

🏭
The Factory Assembly Line Analogy When a factory opens in the morning, the assembly line takes time to warm up — belts moving, robots calibrating. Once warm, each car takes only minutes to assemble. Lambda cold start is the factory warm-up: first invocation loads LangGraph, spaCy, and the Qdrant connection (5–10 seconds). Every subsequent invocation in the same container is fast (under 1 second) because the line is already warm. Always initialise expensive resources at module level, not inside the handler function.
pythonagent/handler.py
import traceback, structlog
from agent.graph import build_graph

log = structlog.get_logger(__name__)

# ── Module-level: runs ONCE per Lambda warm container ───────────
# Moving this inside lambda_handler() = +5-10s on every call
_GRAPH = build_graph()
log.info("agent_graph_ready")

def lambda_handler(event: dict, context) -> dict:
    q = (event.get("question") or "").strip()
    if not q:
        return {"error": "question is required", "answer": None}

    log.info("invoke_start", sid=event.get("session_id"), q=q[:80])
    try:
        state = _GRAPH.invoke({
            "session_id":     event.get("session_id", "anon"),
            "question":       q,
            "location_filter": event.get("location_filter"),
            "messages":        [],
        })
        log.info("invoke_done",
                  intent=state.get("intent"),
                  nodes=state.get("nodes_visited"),
                  alert=state.get("alert_triggered"))
        return {
            "answer":          state["final_answer"],
            "intent":          state.get("intent"),
            "nodes_visited":   state.get("nodes_visited", []),
            "sources":         state.get("sources", []),
            "alert_triggered": state.get("alert_triggered", False),
            "alert_details":   state.get("alert_details", []),
            "error":           None,
        }
    except Exception as e:
        log.error("invoke_failed", error=str(e), tb=traceback.format_exc())
        return {"error": str(e), "answer": None,
                "nodes_visited": [], "intent": None}

🎯 Interview Q&A — Engineering Manager / Senior AI Engineer
Q: Why LangGraph over a simpler LangChain sequential chain for this use case?
Sequential chains work when every query follows the same linear path. Our four intent types require genuinely different tool paths: SUMMARY needs retrieval, TREND needs time-series analytics, COMPARE needs cross-location aggregation, ALERT needs threshold monitoring. LangGraph's conditional routing handles all four in one compiled graph without duplicating code for each path. More importantly, LangGraph's stateful TypedDict gives every node access to everything previous nodes produced — the synthesize node can see the RAG answer AND the analytics rows AND the alert details simultaneously. A sequential chain can't do this cleanly. The added graph complexity pays off immediately when you have more than one tool. Phase 0 locked LangGraph specifically because of this; we knew the agent layer would need multi-path routing from the start.
Q: How do you enforce the Phase 0 safety boundary in the agent layer?
Two independent layers, not one. First, the tool node itself: analytics_query_node uses duckdb.connect(":memory:") pointing at read-only Parquet snapshots from S3 — not a live database connection. Even if the LLM generates a DROP TABLE, DuckDB in-memory mode refuses it at the driver level, not the prompt level. This makes the boundary architectural rather than instructional. Second, the synthesize node's system prompt explicitly forbids SQL and code output, and the eval suite tests this with adversarial prompts. Both layers must hold independently. Prompt-only safety is necessary but not sufficient for production systems where the consequence of a bypass is data modification, not just a bad answer.
Q: How do you measure and maintain agent quality over time?
Three levels. Unit tests on individual node functions with mocked dependencies — tests intent_classifier_node logic without burning API tokens. Integration tests running the full graph against the generated evaluation dataset: 20 questions with known expected intents and expected tool nodes, targeting ≥90% accuracy. Adversarial safety tests with injected SQL, system prompt extraction attempts, and code generation requests. The critical piece is the generated evaluation dataset: FIPAgentGenerator creates questions programmatically from templates with known ground-truth labels. As new edge cases appear, add them to TEMPLATES and re-generate the dataset — the test coverage grows alongside the code. Run all three in CI and block deploys on accuracy drops below 90% or any safety failure.
Q: How would you add multi-turn conversation support without losing the current architecture?
LangGraph supports multi-turn natively via the messages field in AgentState, annotated with the add_messages reducer which accumulates rather than overwrites. For persistence between Lambda invocations (Lambda is stateless), store the AgentState JSON in DynamoDB keyed by session_id. On each new invocation, load the previous state, append the new question, and invoke the graph — the intent classifier has full conversation context. The practical challenge is context window management: after 10+ turns, message history approaches the model token limit. Mitigation: sliding window keeping the last 5 exchanges plus the original question, discarding middle history when approaching the token budget. The architecture doesn't change; you add a state-load step at the start of lambda_handler and a state-save step at the end, both targeting DynamoDB.
