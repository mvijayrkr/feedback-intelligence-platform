Bronze Layer

Kafka Consumer → Bronze Parquet Writer

Exactly-once delivery, Avro deserialization with Schema Registry, idempotent Parquet writes with manual offset commits.

🔥
Production Issue #1: The Auto-Commit Duplicate Trap enable.auto.commit=True is Kafka's default and will cause data loss OR duplicates under any consumer crash. The consumer commits offsets BEFORE your Parquet write completes → crash → reprocess → duplicates in Bronze. Use manual offset commits and commit ONLY after the file is fsynced to storage.
python ingestion/consumers/bronze_writer.py 
"""
Bronze Layer Consumer: Kafka → Parquet (FLOCI S3)
Key guarantee: exactly-once delivery via manual offset commits.
Offsets committed ONLY after Parquet file is durably written.
"""
from __future__ import annotations

import io, time, uuid, signal, sys
from datetime import datetime, timezone
from pathlib import Path
from typing import Iterator
import pyarrow as pa
import pyarrow.parquet as pq
import boto3, structlog
from confluent_kafka import Consumer, KafkaError
from confluent_kafka.schema_registry import SchemaRegistryClient
from confluent_kafka.schema_registry.avro import AvroDeserializer
from confluent_kafka.serialization import SerializationContext, MessageField

log = structlog.get_logger(__name__)

BRONZE_SCHEMA = pa.schema([
    pa.field("event_id",        pa.string(),   nullable=False),
    pa.field("source_platform", pa.string()),
    pa.field("location_id",     pa.string()),
    pa.field("raw_text",        pa.string()),
    pa.field("star_rating",     pa.float32()),
    pa.field("review_date",     pa.timestamp("us", tz="UTC")),
    pa.field("ingested_at",    pa.timestamp("us", tz="UTC")),
    pa.field("kafka_partition", pa.int32()),
    pa.field("kafka_offset",    pa.int64()),
    # Store full raw JSON as string — NEVER lose original data
    pa.field("_raw_json",       pa.string()),
])

class BronzeWriter:
    """
    Accumulates Kafka messages into in-memory buffer,
    writes a Parquet file per batch, then commits offsets.
    Pattern: consume → buffer → write → commit (never auto-commit).
    """
    def __init__(self, config: dict):
        self.config     = config
        self.s3         = boto3.client("s3", **config["aws"])
        self.bucket     = config["bronze_bucket"]
        self.batch_size = config.get("batch_size", 500)  # 500 msgs or 60s
        self.flush_secs = config.get("flush_interval_secs", 60)
        self._buffer: list[dict] = []
        self._last_flush = time.monotonic()
        self._pending_offsets: dict = {}
        self._shutdown = False

        # Schema Registry client for Avro deserialization
        sr_client = SchemaRegistryClient(
            {"url": config["schema_registry_url"]}
        )
        self.deserializer = AvroDeserializer(sr_client)

        self.consumer = Consumer({
            "bootstrap.servers": config["kafka_brokers"],
            "group.id":          "fip-bronze-writer-v2",
            # ─── CRITICAL: disable auto-commit ───────────────────
            "enable.auto.commit": False,
            # ─────────────────────────────────────────────────────
            "auto.offset.reset":  "earliest",
            "max.poll.interval.ms": 300_000,
            "session.timeout.ms":   30_000,
            "fetch.max.bytes":       52_428_800,  # 50 MB
        })

    def _deserialize(self, msg) -> dict | None:
        try:
            return self.deserializer(
                msg.value(),
                SerializationContext(msg.topic(), MessageField.VALUE)
            )
        except Exception as e:
            # Dead-letter queue: write undeserializable messages to DLQ
            self._send_to_dlq(msg, str(e))
            log.error("deserialization_failed",
                       partition=msg.partition(), offset=msg.offset(), error=str(e))
            return None

    def _send_to_dlq(self, msg, reason: str):
        """Route poison-pill messages to DLQ topic, never drop silently."""
        # Producer to fip.dlq.feedback topic — alert on-call if DLQ grows
        key = f"{msg.partition()}:{msg.offset()}"
        log.warning("dlq_sent", key=key, reason=reason)
        # TODO: implement DLQ producer (covered in monitoring phase)

    def _build_row(self, record: dict, msg) -> dict:
        import json
        return {
            "event_id":        record.get("event_id", str(uuid.uuid4())),
            "source_platform": record.get("source_platform", "unknown"),
            "location_id":     record.get("location_id", "unknown"),
            "raw_text":        record.get("review_text", ""),
            "star_rating":     float(record.get("star_rating", 0)),
            "review_date":     datetime.fromisoformat(record["review_date"]),
            "ingested_at":     datetime.now(timezone.utc),
            "kafka_partition": msg.partition(),
            "kafka_offset":    msg.offset(),
            "_raw_json":       json.dumps(record),
        }

    def _flush(self):
        if not self._buffer:
            return

        now = datetime.now(timezone.utc)
        s3_key = (
            f"year={now.year}/month={now.month:02d}"
            f"/day={now.day:02d}/hour={now.hour:02d}"
            f"/batch-{uuid.uuid4().hex[:8]}.parquet"
        )

        try:
            table = pa.Table.from_pylist(self._buffer, schema=BRONZE_SCHEMA)
            buf   = io.BytesIO()
            pq.write_table(
                table, buf,
                compression="snappy",
                row_group_size=100_000,   # ~100 K rows per row group
                write_statistics=True,
            )
            buf.seek(0)
            self.s3.put_object(
                Bucket=self.bucket, Key=s3_key, Body=buf,
                ContentType="application/octet-stream",
            )
            log.info("bronze_flush", key=s3_key, rows=len(self._buffer))

            # ─── Commit offsets ONLY after durable write ─────────
            self.consumer.commit(asynchronous=False)
            # ─────────────────────────────────────────────────────
            self._buffer.clear()
            self._pending_offsets.clear()
            self._last_flush = time.monotonic()

        except Exception as e:
            log.error("flush_failed", s3_key=s3_key, error=str(e))
            # DO NOT commit offsets — let Kafka rewind on restart
            raise  # crash loud, don't swallow errors silently

    def run(self):
        self.consumer.subscribe(["raw.feedback"])
        # Graceful shutdown handler
        signal.signal(signal.SIGTERM, lambda *_: setattr(self, "_shutdown", True))

        while not self._shutdown:
            msg = self.consumer.poll(timeout=1.0)
            if msg is None: pass
            elif msg.error():
                if msg.error().code() != KafkaError._PARTITION_EOF:
                    log.error("kafka_error", error=msg.error())
            else:
                record = self._deserialize(msg)
                if record:
                    self._buffer.append(self._build_row(record, msg))

            # Flush on size OR time threshold
            elapsed = time.monotonic() - self._last_flush
            if len(self._buffer) >= self.batch_size or elapsed >= self.flush_secs:
                self._flush()

        # Graceful drain on shutdown
        self._flush()
        self.consumer.close()
DecisionOption AOption B (Chosen)Reason
Offset commit Auto-commit (risky) Manual after write ✓ Prevents duplicates on crash
Flush trigger Record count only Count OR time limit ✓ Low-volume hours still flush
Error handling Swallow + continue DLQ + re-raise ✓ No silent data loss
Compression GZIP (slow write) Snappy ✓ 2× faster, splittable

Schema Evolution

Avro Schema Registry & Forward/Backward Compatibility

The single most underestimated production problem: a new field from a platform API breaks your entire pipeline at 2 AM. Schema Registry + BACKWARD_TRANSITIVE compatibility is the fix.

🔥
Production Issue #2: Schema Drift Crashes Google Maps adds a new field reviewer_level in their API response. Your Avro deserializer has no field for it → SchemaResolutionException → every consumer crashes → your pipeline stops for hours. BACKWARD compatibility means new schemas can read old data. Register schema changes through the registry, NEVER skip it.

"""Register schema with BACKWARD_TRANSITIVE compatibility check."""
from confluent_kafka.schema_registry import SchemaRegistryClient, Schema

def register_schema(registry_url: str, subject: str, schema_path: str):
    client = SchemaRegistryClient({"url": registry_url})

    # Enforce BACKWARD_TRANSITIVE — new consumers CAN read ALL old messages
    client.set_compatibility(subject, "BACKWARD_TRANSITIVE")

    with open(schema_path) as f:
        schema_str = f.read()

    schema = Schema(schema_str, "AVRO")
    schema_id = client.register_schema(subject, schema)
    print(f"Registered schema ID: {schema_id}")

    # Test compatibility BEFORE registering in prod CI pipeline
    is_compatible = client.test_compatibility(subject, schema)
    if not is_compatible:
        raise ValueError("Schema breaks backward compatibility! CI blocked.")

Fix: cap partition size at 128 MB, set spark.sql.shuffle.partitions to match your data volume, and use broadcast joins for location lookup tables.

"""
Bronze → Silver PySpark Job
Runs: spark-submit --conf spark.executor.memory=4g \
      --conf spark.driver.memory=2g \
      processing/spark_jobs/bronze_to_silver.py
"""
import re, hashlib
from datetime import datetime, timezone
from pyspark.sql import SparkSession
from pyspark.sql import functions as F
from pyspark.sql.types import (
    StructType, StructField, StringType, FloatType,
    TimestampType, ArrayType, IntegerType, BooleanType
)
from pyspark.sql.window import Window

def build_spark() -> SparkSession:
    return (
        SparkSession.builder
        .appName("fip-bronze-to-silver")
        # ─── MEMORY TUNING — tune per your cluster ─────────────
        .config("spark.executor.memory",            "4g")
        .config("spark.executor.memoryOverhead",    "1g")   # JVM overhead
        .config("spark.driver.memory",              "2g")
        .config("spark.sql.shuffle.partitions",     "48")   # 12 locations × 4
        .config("spark.sql.files.maxPartitionBytes", "134217728")  # 128 MB
        # ─── PREVENT SMALL FILE PROBLEM ────────────────────────
        .config("spark.sql.adaptive.enabled",        "true")
        .config("spark.sql.adaptive.coalescePartitions.enabled", "true")
        # ─────────────────────────────────────────────────────
        .config("spark.hadoop.fs.s3a.endpoint", "http://localhost:4566")
        .getOrCreate()
    )

# UDF: PII masking — phone numbers, emails
@F.udf(StringType())
def mask_pii(text: str) -> str:
    if not text: return text
    # Mask phone numbers
    text = re.sub(r'\b\d{3}[-.]?\d{3}[-.]?\d{4}\b', '[PHONE]', text)
    # Mask email addresses
    text = re.sub(r'\b[A-Za-z0-9._%+\-]+@[A-Za-z0-9.\-]+\.[A-Z|a-z]{2,}\b',
                   '[EMAIL]', text)
    return text

# UDF: deterministic dedup ID (event_id + review_date)
@F.udf(StringType())
def make_dedup_key(event_id: str, review_date: str) -> str:
    raw = f"{event_id}|{review_date}"
    return hashlib.sha256(raw.encode()).hexdigest()

def dedup_reviews(df):
    """
    Window-based dedup: keep earliest ingestion of each dedup_key.
    This is idempotent — running twice produces same Silver output.
    TRADEOFF: requires a full sort pass, ~30% slower than dropDuplicates()
              but dropDuplicates() is non-deterministic across reruns.
    """
    w = Window.partitionBy("dedup_key").orderBy(F.col("ingested_at").asc())
    return (
        df.withColumn("_rank", F.row_number().over(w))
          .filter(F.col("_rank") == 1)
          .drop("_rank")
    )

def validate_data_quality(df):
    """Fail-fast validation before writing Silver — no silent bad data."""
    total = df.count()
    null_text = df.filter(F.col("raw_text").isNull() | (F.length("raw_text") < 5)).count()
    bad_rating = df.filter(
        (F.col("star_rating") < 1) | (F.col("star_rating") > 5)
    ).count()

    null_pct  = (null_text  / total) * 100
    bad_pct   = (bad_rating / total) * 100

    if null_pct > 5.0:
        raise ValueError(f"DATA QUALITY GATE FAILED: {null_pct:.1f}% null text (threshold: 5%)")
    if bad_pct > 2.0:
        raise ValueError(f"DATA QUALITY GATE FAILED: {bad_pct:.1f}% bad ratings (threshold: 2%)")

    return df.filter(
        F.col("raw_text").isNotNull() &
        (F.length("raw_text") >= 5) &
        F.col("star_rating").between(1, 5)
    )

def run_bronze_to_silver(spark: SparkSession, run_date: str):
    # Incremental: process ONE partition (Hive-style) per run
    s3_bronze = f"s3a://fip-curry-bowl-local-bronze/year={run_date[:4]}"

    bronze_df = spark.read.parquet(s3_bronze)

    silver_df = (
        bronze_df
        .withColumn("dedup_key",    make_dedup_key("event_id", "review_date"))
        .withColumn("clean_text",   mask_pii(F.col("raw_text")))
        .withColumn("text_length",  F.length("clean_text"))
        .withColumn("processed_at", F.current_timestamp())
        .withColumn("year",          F.year("review_date"))
        .withColumn("month",         F.month("review_date"))
    )

    silver_df = dedup_reviews(silver_df)
    silver_df = validate_data_quality(silver_df)

    # Repartition to fix skew before write
    # 12 locations × 2 = 24 partitions (avoids tiny files)
    silver_df = silver_df.repartition(24, "location_id", "year", "month")

    (silver_df.write
        .mode("overwrite")   # Idempotent: re-run same date → same output
        .partitionBy("year", "month", "location_id")
        .parquet("s3a://fip-curry-bowl-local-silver/")
    )
    print(f"Silver write complete: {silver_df.count()} rows")

if __name__ == "__main__":
    spark = build_spark()
    run_date = str(datetime.now(timezone.utc).date())
    run_bronze_to_silver(spark, run_date)
    spark.stop()

Gold Layer — dbt

dbt Incremental Models: Analytics-Ready Aggregations

Build production dbt incremental models over Silver Parquet. The tradeoff between full-refresh vs incremental, and why late-arriving data breaks naive dbt models.

🔥
Production Issue #5: Late-Arriving Data Breaks Incremental Models Yelp delivers review data up to 72 hours late (API rate limits + batch delivery). A naive dbt incremental model with is_incremental() checks MAX(processed_at) from last run. The 3-day-old review arrives → it's before the watermark → SILENTLY DROPPED. Fix: use a lookback window of 4 days in your incremental filter, not the exact MAX timestamp.

-- Gold: daily location-level rating aggregations
-- Incremental with 4-day lookback to catch late-arriving Yelp data

{{
  config(
    materialized     = 'incremental',
    unique_key       = ['location_id', 'review_date'],
    on_schema_change = 'append_new_columns',
    -- CRITICAL: use partition_by for large tables
    partition_by = {
      'field': 'review_date',
      'data_type': 'date',
      'granularity': 'month'
    }
  )
}}

WITH source_reviews AS (
    SELECT
        event_id,
        location_id,
        CAST(review_date AS DATE) AS review_date,
        star_rating,
        clean_text,
        text_length,
        source_platform,
        processed_at
    FROM {{ source('silver', 'reviews') }}

    {% if is_incremental() %}
    WHERE
        -- 4-day lookback window: catches Yelp's 72h late delivery
        -- NEVER use: WHERE processed_at > (SELECT MAX(processed_at) FROM {{ this }})
        processed_at >= CURRENT_TIMESTAMP() - INTERVAL '4 days'
    {% endif %}
),

aggregated AS (
    SELECT
        location_id,
        review_date,
        COUNT(*)                               AS review_count,
        ROUND(AVG(star_rating), 2)             AS avg_rating,
        SUM(CASE WHEN star_rating >= 4 THEN 1 ELSE 0 END) AS positive_count,
        SUM(CASE WHEN star_rating <= 2 THEN 1 ELSE 0 END) AS negative_count,
        ROUND(AVG(text_length), 0)             AS avg_review_length,
        COUNT(DISTINCT source_platform)         AS platform_count,
        ARRAY_AGG(DISTINCT source_platform)    AS platforms,
        MAX(processed_at)                      AS last_updated_at
    FROM source_reviews
    GROUP BY location_id, review_date
)

SELECT
    *,
    ROUND(positive_count::float / NULLIF(review_count, 0) * 100, 1) AS positive_pct,
    ROUND(negative_count::float / NULLIF(review_count, 0) * 100, 1) AS negative_pct
FROM aggregated

Great Expectations: Quality Gates & Anomaly Detection

Production data quality is an SLA, not a nice-to-have. GX suites run as Airflow tasks — pipeline halts on failure, Slack alert fires, on-call is paged.

⚠️
Production Issue #6: Silent Bad Data Flowing to RAG If corrupted Silver data flows into Qdrant embeddings (Phase 4), the RAG responses will confidently hallucinate business insights from garbage. A Great Expectations gate between Silver and Gold is the circuit breaker. Fail loudly here rather than generate wrong business decisions downstream.

"""
Great Expectations validation suite for Silver layer.
Run as Airflow PythonOperator BEFORE triggering dbt Gold run.
If this raises, the DAG stops — Gold never sees bad data.
"""
import great_expectations as gx
from great_expectations.core.batch import RuntimeBatchRequest

def build_silver_suite(context):
    suite = context.add_or_update_expectation_suite("fip.silver.reviews")

    validator = context.get_validator(
        batch_request=RuntimeBatchRequest(...),
        expectation_suite=suite
    )

    # ─── Completeness ──────────────────────────────────────
    validator.expect_column_values_to_not_be_null("event_id")
    validator.expect_column_values_to_not_be_null("location_id")
    validator.expect_column_values_to_not_be_null("clean_text")

    # ─── Validity ──────────────────────────────────────────
    validator.expect_column_values_to_be_between(
        "star_rating", min_value=1.0, max_value=5.0
    )
    validator.expect_column_values_to_be_in_set(
        "source_platform",
        {"google", "yelp", "doordash", "ubereats", "tripadvisor"}
    )
    validator.expect_column_value_lengths_to_be_between(
        "clean_text", min_value=5, max_value=10000
    )

    # ─── Uniqueness ────────────────────────────────────────
    validator.expect_column_values_to_be_unique("dedup_key")

    # ─── Volume anomaly detection ──────────────────────────
    # Alert if review volume drops below 50/day (ingestion broken?)
    validator.expect_table_row_count_to_be_between(
        min_value=50, max_value=100_000
    )

    # ─── Distribution checks ───────────────────────────────
    validator.expect_column_mean_to_be_between(
        "star_rating", min_value=3.0, max_value=5.0
        # If average rating suddenly drops to 1.2 → data pipeline bug or real crisis
    )
    validator.expect_column_proportion_of_unique_values_to_be_between(
        "location_id", min_value=0.001, max_value=1.0
    )

    validator.save_expectation_suite(discard_failed_expectations=False)
    results = validator.validate()

    if not results.success:
        failed = [r for r in results.results if not r.success]
        raise ValueError(
            f"Silver data quality gate FAILED: {len(failed)} checks failed.\n"
            + "\n".join(str(r.expectation_config) for r in failed)
        )

    return results

Vector Store Bootstrap

Qdrant Collection Schema & Gold → Vector Backfill

Bootstrap Qdrant collections with correct schema, distance metrics, and HNSW index parameters before Phase 4 (RAG) inserts embeddings. Get this wrong and you re-index everything.

🔥
Production Issue #7: Qdrant Collection Versioning & Re-indexing Cost You cannot change the vector dimension or distance metric of an existing Qdrant collection. If Phase 4 decides to switch from OpenAI text-embedding-3-small (1536-dim) to text-embedding-3-large (3072-dim), you must create a new collection and re-embed ALL data. At $0.00002/1K tokens × 50K reviews × 200 tokens avg = $0.20 per re-embed. Small now, painful at 5M reviews. Lock the dimension in Phase 2, don't defer it.
⚠️
Production Issue #8: HNSW ef_construct Tradeoff Higher ef_construct = better recall but slower indexing. At 50K vectors, ef_construct=200 takes ~2 min to build. At 5M vectors, it takes 3+ hours. Start at 128, benchmark recall on your data, only increase if RAG answers are bad. Do NOT blindly copy defaults from documentation examples that use 10K toy datasets.
python qdrant/bootstrap/create_collections.py 
"""
Qdrant collection bootstrap — run ONCE before Phase 4 (RAG).
Creates versioned collections so we can blue/green swap embeddings.
"""
from qdrant_client import QdrantClient, models
from qdrant_client.models import (
    VectorParams, Distance, HnswConfigDiff,
    OptimizersConfigDiff, PayloadSchemaType
)
import structlog

log = structlog.get_logger(__name__)

# ── Collection versioning strategy ─────────────────────────────
# Never mutate a live collection. Use blue/green:
# fip_reviews_v1 (current) → fip_reviews_v2 (new model) → swap alias
COLLECTION_NAME = "fip_reviews_v1"
EMBEDDING_DIM   = 1536   # text-embedding-3-small — LOCKED. Changing = re-index all.
DISTANCE_METRIC = Distance.COSINE  # Cosine for semantic similarity (normalized embeddings)

def bootstrap_qdrant(client: QdrantClient) -> None:
    existing = {c.name for c in client.get_collections().collections}

    if COLLECTION_NAME in existing:
        log.info("collection_exists", name=COLLECTION_NAME)
        return

    client.create_collection(
        collection_name=COLLECTION_NAME,
        vectors_config=VectorParams(
            size=EMBEDDING_DIM,
            distance=DISTANCE_METRIC,
            # on_disk=True saves RAM at the cost of ~20% query slowdown
            # Use on_disk=False for < 1M vectors (fits comfortably in 8 GB RAM)
            on_disk=False,
        ),
        hnsw_config=HnswConfigDiff(
            m=16,           # Number of bi-directional links per node
                            # Higher M = better recall, more memory (m=16 is good default)
            ef_construct=128,  # Build-time search width (higher = slower build, better recall)
            full_scan_threshold=10_000,  # Below this, use brute-force (always 100% recall)
        ),
        optimizers_config=OptimizersConfigDiff(
            indexing_threshold=20_000,  # Start HNSW indexing after 20K vectors
            memmap_threshold=50_000,   # Memory-map segments above 50K vectors
        ),
    )

    # ── Payload indexes for filtered hybrid search ──────────────
    # ALWAYS index fields you'll filter on. Without this, every filter
    # does a full scan = O(N) instead of O(1).
    payload_indexes = {
        "location_id":     PayloadSchemaType.KEYWORD,
        "source_platform": PayloadSchemaType.KEYWORD,
        "star_rating":     PayloadSchemaType.FLOAT,
        "review_date":     PayloadSchemaType.DATETIME,
        "year":            PayloadSchemaType.INTEGER,
        "month":           PayloadSchemaType.INTEGER,
    }

    for field, schema_type in payload_indexes.items():
        client.create_payload_index(
            collection_name=COLLECTION_NAME,
            field_name=field,
            field_schema=schema_type,
        )
        log.info("payload_index_created", field=field)

    # ── Alias for blue/green swap ───────────────────────────────
    client.update_collection_aliases(
        change_aliases_operations=[
            models.CreateAliasOperation(
                create_alias=models.CreateAlias(
                    collection_name=COLLECTION_NAME,
                    alias_name="fip_reviews_current",   # RAG always queries this alias
                )
            )
        ]
    )
    log.info("qdrant_bootstrap_complete", collection=COLLECTION_NAME)


if __name__ == "__main__":
    client = QdrantClient(host="localhost", port=6333)
    bootstrap_qdrant(client)

"""Airflow DAG: FIP Data Platform — runs nightly at 2 AM PT"""
from datetime import datetime, timedelta
from airflow import DAG
from airflow.operators.python import PythonOperator
from airflow.providers.apache.spark.operators.spark_submit import SparkSubmitOperator
from airflow.providers.slack.operators.slack_webhook import SlackWebhookOperator

def on_failure_callback(context):
    return SlackWebhookOperator(
        task_id="slack_alert",
        webhook_token="{{ var.value.slack_webhook }}",
        message=(
            f":red_circle: FIP Pipeline FAILED\n"
            f"Task: {context['task_instance'].task_id}\n"
            f"DAG: {context['dag'].dag_id}\n"
            f"Log: {context['task_instance'].log_url}"
        )
    ).execute(context)

with DAG(
    dag_id="fip_data_platform",
    schedule_interval="0 10 * * *",   # 2 AM PT = 10 UTC
    start_date=datetime(2025, 1, 1),
    default_args={
        "retries":           2,
        "retry_delay":       timedelta(minutes=10),
        "on_failure_callback": on_failure_callback,
        "execution_timeout": timedelta(hours=3),
    },
    catchup=False,
    tags=["fip", "data-platform"],
) as dag:

    spark_bronze_to_silver = SparkSubmitOperator(
        task_id="spark_bronze_to_silver",
        application="processing/spark_jobs/bronze_to_silver.py",
        conf={
            "spark.executor.memory": "4g",
            "spark.executor.memoryOverhead": "1g",
        },
        application_args=["--run-date", "{{ ds }}"],
    )

    gx_silver_validation = PythonOperator(
        task_id="gx_silver_validation",
        python_callable=lambda: validate_silver_suite("{{ ds }}"),
    )

    dbt_gold_run = PythonOperator(
        task_id="dbt_gold_run",
        python_callable=lambda: run_dbt_command("dbt run --select marts"),
    )

    dbt_gold_test = PythonOperator(
        task_id="dbt_gold_test",
        python_callable=lambda: run_dbt_command("dbt test --select marts"),
    )

    qdrant_verify = PythonOperator(
        task_id="qdrant_collection_verify",
        python_callable=lambda: verify_qdrant_health(),
    )

    # ── Pipeline dependency chain ──────────────────────────────
    # Bronze → Silver → [GX gate] → Gold run → Gold test → Qdrant verify
    # GX gate is a hard stop — Gold never runs on bad Silver data
    (spark_bronze_to_silver
        >> gx_silver_validation
        >> dbt_gold_run
        >> dbt_gold_test
        >> qdrant_verify)

Here are 3 things that would have burned us at 3 AM if we hadn't caught them in design: 🔥 The auto-commit trap Kafka's default enable.auto.commit=True commits offsets BEFORE your Parquet file is written to S3. 🎯 Interview Q&A — Senior Data/AI Engineer
Q: How do you guarantee exactly-once delivery in your Kafka pipeline?
We use manual offset commits on the consumer side: offsets are only committed after the Parquet file is confirmed durably written to S3. This gives us at-least-once delivery (possible duplicates on crash) which we handle at the Silver layer with idempotent window-based deduplication on a SHA-256 dedup key. True exactly-once at scale requires either Kafka transactions (high latency) or this two-phase approach. We chose the latter because the Bronze layer is append-only and re-runnable.
Q: How do you handle schema evolution in a streaming pipeline without downtime?
We use Confluent Schema Registry with BACKWARD_TRANSITIVE compatibility enforced at the subject level. Any new field added to the Avro schema must have a default value (null for optional fields). We block PRs that break compatibility via a CI step that calls test_compatibility() against the registry before the schema is merged. This means old consumers can still read new messages, and we can roll out producers before consumers without coordinating a big-bang deploy.
Q: What's your strategy for dbt incremental models with late-arriving data?
The naive approach — filtering WHERE processed_at > MAX(processed_at in the existing table) — silently drops any records delivered after that watermark. For our pipeline, Yelp delivers data up to 72 hours late. We use a 4-day lookback window instead of an exact watermark, and we pair this with a weekly full-refresh run (scheduled Sunday 3 AM) to self-correct any edge cases. The tradeoff is slightly higher compute cost on incremental runs, but that's a rounding error compared to the cost of silently wrong analytics data.
Q: How do you handle Spark OOM errors on variable-length text data?
Three levers: (1) Set spark.sql.files.maxPartitionBytes to 128 MB to prevent oversized input partitions, (2) Configure executor memory with an explicit overhead budget — 4g executor + 1g overhead for JVM — because Spark memory is separate from JVM native memory, (3) Enable Adaptive Query Execution which coalesces small partitions and splits large ones at runtime. For text-heavy workloads, string data can be 5× larger than you estimate from row counts alone — always size memory based on your P99 review length, not average.
