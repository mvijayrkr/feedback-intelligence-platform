01

Core Build Principles

The rules that keep all phase guides synchronized.

🏗️ Incremental Only

Each phase consumes the outputs of the previous phase and adds new capabilities. Nothing gets recreated unless explicitly destroyed.

☁️ FLOCI First

Phase 0 starts FLOCI and provisions AWS-shaped resources. Phases 1–5 deploy into that environment. Phase 6 promotes the same model to AWS.

🚢 Terraform + Helm

Terraform owns infrastructure. Helm owns Kubernetes deployment. Docker Compose is not the main runtime path for the project.

✅
Continuity Contract The execution flow is always make bootstrap once, then make phase1, make phase2, make phase3, make phase4, make phase5, and finally make phase6.

⚠️
What not to do Do not create a new repository in later phases. Do not run local Docker Compose Kafka as the main path. Do not recreate S3/RDS/MSK. Later phases extend Terraform, Helm, services, dbt, and app folders inside the same monorepo.

Engineering Manager Interview Lens

How to tell the story in leadership, system design, and AI platform interviews.

🎯 Story Pattern

“I did not build seven disconnected demos. I built one evolving platform. Phase 0 gave the team AWS-parity locally. Every phase added a production capability without breaking deployment discipline.”

⚖️ Tradeoff Pattern

“The tradeoff was initial setup time versus future velocity. FLOCI + Terraform + Helm cost more on day one, but it avoided local/prod drift and made production promotion predictable.”

🧠 AI Safety Pattern

“Agents never query arbitrary tables directly. They use governed tools over Gold marts and tenant-filtered RAG retrieval, with audit logs for every tool call.”

📈 Scale Pattern

“The system scales independently: Kafka for ingestion, S3 for lake storage, dbt for marts, Qdrant for search, LangGraph for orchestration, and React/FastAPI for UX.”
