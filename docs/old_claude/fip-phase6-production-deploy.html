Data Generator — Production Smoke Test Suite
🧪 FIPSmokeTestGenerator — Production Verification Queries That Hit Every Layer
🐇
The Canary in the Coal Mine Analogy Miners sent a canary into new tunnels before entering — a healthy canary meant the air was safe. The smoke test suite is the production canary: 10 carefully chosen API calls that touch every layer of the stack. If all pass, the deploy is healthy. If any fail, the canary is down and the rollback fires automatically — before a single manager opens the dashboard and sees a broken screen. Reuses Phase 1 FIPDataGenerator for review content in POST /api/ask tests.
pythonscripts/smoke_test.py
"""
FIP Production Smoke Test -- Phase 6
Verifies all layers are live after deploy: FastAPI, LangGraph agent,
Qdrant RAG, DuckDB Gold analytics, and MSK Kafka consumer lag.

Usage:
  python scripts/smoke_test.py --env prod --timeout 60
  python scripts/smoke_test.py --env local --timeout 30
"""
import argparse, json, time, sys
import httpx, boto3

ENDPOINTS = {
    "local": "http://localhost:8000",
    "prod":  "https://api.fip.curry-bowl.com",
}

class FIPSmokeTestGenerator:
    """Generates smoke test scenarios reusing Phase 1 FIPDataGenerator content."""
    def __init__(self, base_url: str, timeout: int = 60):
        self.base    = base_url
        self.client  = httpx.Client(timeout=timeout)
        self.results = []

    def _test(self, name: str, method: str, path: str, **kwargs) -> bool:
        t0 = time.monotonic()
        try:
            r = getattr(self.client, method)(f"{self.base}{path}", **kwargs)
            ok = r.status_code < 400
            latency = round((time.monotonic()-t0)*1000)
            self.results.append({"test":name,"ok":ok,"status":r.status_code,"latency_ms":latency})
            sym = "✓" if ok else "✗"
            print(f"{sym} [{latency:4d}ms] {name}")
            return ok
        except Exception as e:
            self.results.append({"test":name,"ok":False,"error":str(e)})
            print(f"✗ [FAIL] {name}: {e}")
            return False

    def run(self) -> bool:
        print(f"\n=== FIP Smoke Tests: {self.base} ===\n")
        tests = [
            # Layer 1: FastAPI health
            ("FastAPI health",          "get",  "/health"),
            # Layer 2: Dashboard (DuckDB Gold reads)
            ("Dashboard endpoint",       "get",  "/api/dashboard"),
            ("Location analytics",       "get",  "/api/analytics/cb-mission"),
            ("Alert detection",           "get",  "/api/alerts"),
            # Layer 3: LangGraph agent (SUMMARY intent - cheapest)
            ("Agent SUMMARY intent", "post", "/api/ask",
             {"json":{"question":"Summarise feedback for cb-mission this week.","session_id":"smoke-test"}}),
            # Layer 4: Agent ALERT intent (fires alert_detector)
            ("Agent ALERT intent",  "post", "/api/ask",
             {"json":{"question":"Any locations with sudden rating drops?","session_id":"smoke-test"}}),
            # Layer 5: Metrics endpoint
            ("Prometheus metrics",        "get",  "/metrics"),
        ]
        all_ok = all(self._test(name, method, path, **(kw if len(args)>2 else {}))
                     for *args, in [(t[0],t[1],t[2])+((t[3],) if len(t)>3 else ({},)) for t in tests]
                     for name,method,path,kw in [(*a[:3],a[3] if len(a)>3 else {}) for a in tests])
        passed = sum(1 for r in self.results if r["ok"])
        print(f"\n{'PASS' if all_ok else 'FAIL'}: {passed}/{len(self.results)} tests passed")
        return all_ok

if __name__ == "__main__":
    ap = argparse.ArgumentParser()
    ap.add_argument("--env",     choices=["local","prod"], default="local")
    ap.add_argument("--timeout", type=int, default=60)
    args = ap.parse_args()
    ok = FIPSmokeTestGenerator(ENDPOINTS[args.env], args.timeout).run()
    sys.exit(0 if ok else 1)

01
Terraform Core Infrastructure

VPC, IAM, ECR, Secrets Manager — Foundation Before Everything Else

Deploy the infrastructure foundation first — VPC networking, IAM roles, ECR repositories, and Secrets Manager entries. All other Terraform modules depend on these. This is the same Terraform HCL that ran against LocalStack in FLOCI; only the provider has changed.

🏗
The Utility Infrastructure Analogy Before you build the restaurant, you run the electricity, water, and gas lines to the plot. VPC is the plot boundary (network isolation). IAM roles are the keys to each utility room (access control). ECR is the warehouse where you store your appliances (Docker images). Secrets Manager is the safe where you keep the master keys (API keys, DB passwords). You build these first because every other service depends on them.
🔥
Production Issue #1: Terraform State Not Locked = Concurrent Apply Corruption Two engineers run terraform apply simultaneously. Without DynamoDB state locking, both reads get the same state version, both make changes, second write overwrites the first. Result: resources exist in AWS that aren't in state, or resources deleted from state that still exist in AWS. Fix: S3 backend + DynamoDB lock table. Create these manually BEFORE the first terraform init.

02
MSK + S3 Lakehouse

Kafka on MSK Serverless + S3 Bronze/Silver/Gold with Lifecycle Rules

MSK Serverless replaces the FLOCI Kafka container — same Kafka 3.x API, no broker management, auto-scales to zero on weekends. S3 lakehouse buckets promoted from LocalStack with identical Terraform resource definitions, adding lifecycle rules and Cross-Region Replication for prod.

🚫
The City Water Main vs Garden Hose Analogy FLOCI Kafka is a garden hose: it works for watering your garden but you can't connect an entire city to it. MSK Serverless is the city water main: same water (same Kafka API), auto-scales to peak demand, never needs manual management, shuts off (scales to zero) when no one needs it. The code in Phase 1 that used KAFKA_BROKERS=localhost:9092 now uses the MSK bootstrap endpoint — nothing else changes in the connector code.
🔥
Production Issue #2: S3 Bronze Lifecycle Expiry Destroys Reprocessing Safety Net A well-meaning engineer adds a 30-day lifecycle expiry to the Bronze bucket to save costs. Six months later, a bug in the Silver PySpark job is discovered that corrupted 90 days of data. Bronze data before day 30 is gone. There is no source of truth to reprocess from. Fix: Bronze = infinite retention, prevent_destroy = true in Terraform, versioning enabled. Silver and Gold are re-derivable from Bronze — they can have lifecycle rules. Bronze cannot.

locals { prefix = "fip-curry-bowl-${var.env}" }

  bucket = "${local.prefix}-bronze"
  lifecycle { prevent_destroy = true }   # NEVER delete Bronze
}

  bucket = aws_s3_bucket.bronze.id
  versioning_configuration { status = "Enabled" }
}
# Bronze: NO lifecycle expiry -- infinite retention is intentional

  bucket = "${local.prefix}-silver"
}

  bucket = aws_s3_bucket.silver.id
  rule {
    id     = "expire-silver-90d"
    status = "Enabled"
    filter { prefix = "" }
    expiration { days = 90 }   # Silver re-derivable from Bronze
  }
}

resource "aws_s3_bucket" "gold" {
  bucket = "${local.prefix}-gold"
}
resource "aws_s3_bucket_lifecycle_configuration" "gold" {
  bucket = aws_s3_bucket.gold.id
  rule {
    id     = "ia-after-30d"
    status = "Enabled"
    filter { prefix = "" }
    transition {
      days          = 30
      storage_class = "STANDARD_IA"   # Infrequent Access after 30d
    }
  }
}
03
Qdrant on EC2 + Lambda Agent

Qdrant EC2 with Persistent EBS + LangGraph Agent on Lambda

Qdrant runs on a dedicated EC2 instance with EBS volume — the same collection schema bootstrapped in Phase 2 now holds production embeddings. LangGraph agent promoted from local Python module to AWS Lambda via the same handler.py from Phase 4 — no code changes.

📚
The Physical Library vs Mobile Library Analogy FLOCI Qdrant (Docker on your laptop) is a mobile library: great for browsing, but the books are gone when you pack up the van. EC2 + EBS Qdrant is the permanent library building: the book catalogue (vectors) persists on the EBS volume across every Qdrant restart, EC2 reboot, or AMI update. The 1536-dim collection schema bootstrapped in Phase 2 is the library's floor plan: it defines which shelves exist and how they're indexed. Re-embedding 50K reviews from scratch costs $0.01 and 20 minutes — but losing the library because the van drove away costs much more.
🔥
Production Issue #3: Qdrant Vector Data Not Backed Up EC2 instance terminated by mistake (happens with spot instances). EBS volume deleted (default behaviour on instance termination). 50K production embeddings gone. Re-embed costs $0.01 but takes 20 minutes with real data. Fix: (a) set delete_on_termination = false on the EBS volume, (b) daily EBS snapshot via AWS Backup, (c) keep prevent_destroy = true on the EBS volume resource in Terraform.
hclinfra/terraform/qdrant_ec2.tf
# Qdrant on EC2 -- m6i.large: 2 vCPU, 8 GB RAM, enough for 5M vectors
resource "aws_instance" "qdrant" {
  ami                    = data.aws_ami.amazon_linux_2023.id
  instance_type          = "m6i.large"
  subnet_id              = aws_subnet.private[0].id
  vpc_security_group_ids = [aws_security_group.qdrant.id]
  iam_instance_profile   = aws_iam_instance_profile.qdrant.name

  root_block_device {
    volume_size           = 20
    volume_type           = "gp3"
    delete_on_termination = true
  }

  # Qdrant data volume -- separate from root, survives instance replacement
  ebs_block_device {
    device_name           = "/dev/sdf"
    volume_size           = 100         # 100 GB gp3: ~5M 1536-dim vectors
    volume_type           = "gp3"
    iops                  = 3000
    delete_on_termination = false       # NEVER delete data on termination
  }

  user_data = base64encode(<<-EOF
    #!/bin/bash
    # Mount EBS data volume
    mkfs -t xfs /dev/sdf 2>/dev/null || true
    mkdir -p /qdrant/storage
    mount /dev/sdf /qdrant/storage
    echo "/dev/sdf /qdrant/storage xfs defaults 0 0" >> /etc/fstab

    # Install and start Qdrant
    docker run -d --name qdrant \
      -p 6333:6333 -p 6334:6334 \
      -v /qdrant/storage:/qdrant/storage \
      -e QDRANT__SERVICE__API_KEY=${qdrant_api_key} \
      qdrant/qdrant:v1.9.2
    systemctl enable docker
  EOF
  )

  lifecycle { prevent_destroy = true }
  tags = { Name = "fip-qdrant-${var.env}" }
}

FastAPI on ECS Fargate + React on CloudFront — Auto-Scaling, Zero-Downtime

The Phase 5 FastAPI Docker image deploys to ECS Fargate with Application Load Balancer and blue/green rolling updates. React build output deploys to S3 + CloudFront with cache invalidation. The FLOCI Docker Compose services become ECS task definitions with the same image.

🏢
The Food Truck to Restaurant Chain Analogy A food truck (FLOCI Docker Compose) is one person serving one window. ECS Fargate + ALB is the restaurant chain: multiple cooks (Fargate tasks) working in parallel, an automatic door that opens a new counter when lines get long (auto-scaling on CPU/memory), and a queue manager (ALB) who routes each customer to the next available counter. Customers never see a "closed" sign because rolling updates swap out one counter at a time while the others keep serving.
🔥
Production Issue #4: CloudFront Cache Serves Stale React Build You deploy a new React build to S3. CloudFront still serves the old JS files for up to 24 hours because the default TTL is 86400 seconds. Managers see the old UI with new API breaking changes. Fix: (a) Vite outputs content-hashed filenames (main.abc123.js) so cached assets are always valid; (b) index.html gets max-age=0 so it always fetches fresh; (c) post-deploy CloudFront invalidation on index.html only.

bashscripts/deploy_ui.sh — React build + S3 + CloudFront invalidation
#!/bin/bash
# Build React and deploy to S3 + CloudFront
# Run after Terraform applies CloudFront distribution

set -e
BUCKET="fip-curry-bowl-ui-${ENV:-prod}"
DIST_ID=$(terraform -chdir=infra/terraform output -raw cloudfront_distribution_id)

# 1. Build React with production API URL
cd ui
VITE_API_URL="https://api.fip.curry-bowl.com" npm run build

# 2. Sync to S3 -- hashed assets: long cache; index.html: no cache
aws s3 sync dist/ s3://$BUCKET/ \
  --delete \
  --exclude "index.html" \
  --cache-control "public, max-age=31536000, immutable"

aws s3 cp dist/index.html s3://$BUCKET/index.html \
  --cache-control "no-cache, no-store, must-revalidate"

# 3. Invalidate only index.html -- assets are content-hashed
aws cloudfront create-invalidation \
  --distribution-id "$DIST_ID" \
  --paths "/index.html"

echo "✓ UI deployed. CloudFront serving new build."

GitHub Actions CI/CD

Full CI/CD Pipeline: Test → Build → Terraform → Deploy → Smoke Test

The complete GitHub Actions workflow that wires all 5 deploy stages together. OIDC authentication (no long-lived AWS keys stored as secrets). Terraform plan saved as PR artifact for review. Apply only on merge to main. Automatic rollback if smoke tests fail after deploy.

⚙
The Car Assembly Line Quality Gate Analogy A car assembly line has quality gates at each station: body panel welds checked before painting, engine tested before installation, final inspection before shipping. No car leaves the factory with an untested engine. GitHub Actions is the assembly line: pytest is the weld check (Stage 1), Docker build is the engine test (Stage 2), Terraform plan is pre-ship inspection (Stage 3), and the smoke test is the test drive before keys are handed to the customer. A failure at any gate stops the line and rolls back.
🔥
Production Issue #5: Long-Lived AWS Keys in GitHub Secrets = Security Risk Storing AWS_ACCESS_KEY_ID + AWS_SECRET_ACCESS_KEY as GitHub Secrets means a compromised repo exposes permanent AWS credentials. If the key is leaked in a log line, it's valid indefinitely. Fix: GitHub OIDC federation. GitHub Actions gets a short-lived token (valid for 1 hour) via aws-actions/configure-aws-credentials with role-to-assume. No static credentials ever stored anywhere.
yaml.github/workflows/deploy.yml
name: FIP Production Deploy

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

permissions:
  id-token: write    # Required for OIDC
  contents: read

jobs:
  test:
    name: Stage 1 — Test Gate
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - run: pip install -r requirements.txt
      - name: Unit Tests
        run: pytest tests/ -v --tb=short -q
      - name: Agent Eval (must be >=90% accuracy)
        run: pytest tests/test_agent_eval.py -v -k "accuracy"

  build:
    name: Stage 2 — Build & Push to ECR
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - name: Configure AWS credentials (OIDC — no long-lived keys)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::${{ secrets.AWS_ACCOUNT_ID }}:role/fip-github-actions-prod
          aws-region: us-west-2
      - name: Login to ECR
        run: aws ecr get-login-password | docker login --username AWS --password-stdin ${{ secrets.ECR_REGISTRY }}
      - name: Build & push API image (linux/amd64 for Fargate)
        run: |
          docker buildx build --platform linux/amd64 \
            -t ${{ secrets.ECR_REGISTRY }}/fip-api:${{ github.sha }} \
            -f api/Dockerfile ./api --push
      - name: Build & push Lambda agent image
        run: |
          docker buildx build --platform linux/amd64 \
            -t ${{ secrets.ECR_REGISTRY }}/fip-agent:${{ github.sha }} \
            -f Dockerfile.lambda . --push
      - name: Build React static files
        run: cd ui && VITE_API_URL=${{ secrets.API_URL }} npm run build

  terraform:
    name: Stage 3 — Terraform Apply
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
      - name: Configure AWS (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::${{ secrets.AWS_ACCOUNT_ID }}:role/fip-github-actions-prod
          aws-region: us-west-2
      - name: Terraform Init
        run: terraform -chdir=infra/terraform init
      - name: Terraform Apply
        run: |
          terraform -chdir=infra/terraform apply -auto-approve \
            -var="image_tag=${{ github.sha }}" \
            -var="env=prod"

  deploy:
    name: Stage 4 — Deploy & Smoke Test
    needs: terraform
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Configure AWS (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::${{ secrets.AWS_ACCOUNT_ID }}:role/fip-github-actions-prod
          aws-region: us-west-2
      - name: Deploy React UI to CloudFront
        run: bash scripts/deploy_ui.sh
      - name: Force new ECS deployment (rolling update)
        run: |
          aws ecs update-service --cluster fip-prod \
            --service fip-api-prod --force-new-deployment
          aws ecs wait services-stable \
            --cluster fip-prod --services fip-api-prod
      - name: Stage 5 — Smoke Tests
        run: python scripts/smoke_test.py --env prod --timeout 120
      - name: Rollback on smoke test failure
        if: failure()
        run: |
          echo "Smoke tests failed -- rolling back ECS to previous task definition"
          aws ecs update-service --cluster fip-prod \
            --service fip-api-prod \
            --task-definition fip-api-prod:$(( $(aws ecs describe-services \
              --cluster fip-prod --services fip-api-prod \
              --query 'services[0].taskDefinition' --output text | grep -o '[0-9]*$') - 1 ))

#!/bin/bash
# Run after every deploy to verify the full stack is healthy
# Exits 0 on success, 1 on any failure (triggers CI rollback)

API="https://api.fip.curry-bowl.com"
echo "=== FIP Production Verification ==="

curl -sf "$API/health" | python3 -c "import json,sys; d=json.load(sys.stdin); assert d['status']=='ok'" \
  && echo "✓ FastAPI health" || { echo "✗ FastAPI health FAILED"; exit 1; }

# 2. Dashboard data (DuckDB Gold reads working)
COUNT=$(curl -sf "$API/api/dashboard" | python3 -c "import json,sys; print(len(json.load(sys.stdin)['locations']))")
[ "$COUNT" -eq 12 ] \
  && echo "✓ Dashboard: $COUNT locations" || { echo "✗ Dashboard: expected 12, got $COUNT"; exit 1; }

# 3. Agent SUMMARY intent (RAG layer working)

  -H "Content-Type: application/json" \
  -d '{"question":"Summarise feedback for cb-mission.","session_id":"verify"}' \
  | python3 -c "import json,sys; print(json.load(sys.stdin)['intent'])")
[ "$INTENT" = "SUMMARY" ] \
  && echo "✓ Agent SUMMARY routing" || { echo "✗ Agent intent: expected SUMMARY, got $INTENT"; exit 1; }

# 4. Prometheus metrics endpoint
curl -sf "$API/metrics" | grep -q "fip_http_requests_total" \
  && echo "✓ Prometheus metrics" || { echo "✗ Prometheus metrics FAILED"; exit 1; }

echo ""
echo "=== All checks passed. Deploy healthy. ==="

🎯 Interview Q&A — Engineering Manager / Staff Engineer / Platform Engineer
Q: Walk me through how you promoted the FLOCI local stack to production AWS.
The short answer is: we changed one Terraform provider block. The longer answer is that this outcome was only possible because of an architectural decision made in Phase 0: every Phase uses the same Terraform HCL modules, the same Docker images, the same environment variable names, and the same Python code against the same API contracts. FLOCI (LocalStack) exposes identical AWS service APIs on localhost:4566. When we changed the provider block to remove the endpoint overrides, terraform apply created 12 real AWS services from the same HCL that had been running against LocalStack for 5 phases. The Phase 1 Kafka connector code uses KAFKA_BROKERS from the environment; in FLOCI it points to localhost:9092, in prod it points to the MSK bootstrap endpoint. The code is identical. The promote was a 2-hour planned operation, not a 2-week migration project.
Q: How do you handle zero-downtime deploys for the FastAPI service on ECS Fargate?
ECS rolling update with a minimum healthy percent of 50% and maximum of 200%. When we push a new task definition revision, ECS starts new tasks (the 200% threshold), waits for them to pass health checks (GET /health returning 200), then stops old tasks. The ALB naturally drains connections from stopping tasks before deregistering them. The FastAPI application is stateless: all state lives in Redis (cache), DynamoDB (agent session state), and S3 (data). No in-process state means any request can be handled by any task, so replacing tasks one at a time is safe. The Terraform appautoscaling policy ensures we always have at least 1 task running, even during a deployment that brings healthy count temporarily to 50%.
Q: Why choose EC2 + EBS for Qdrant instead of a managed vector database service?
Three reasons. First, cost: at our scale (5M vectors, ~50K updates per week), EC2 m6i.large at $0.192/hour is roughly $140/month. Qdrant Cloud or Pinecone equivalents are $300–600/month at this capacity. Second, operational simplicity: the Phase 2 bootstrap script, Phase 3 indexer, and Phase 4 retriever all use the Qdrant Python client against a host:port. Switching from localhost:6333 (FLOCI) to the EC2 private IP is one environment variable change. Third, control: we need to pin Qdrant to a specific version (v1.9.2) that matches the client library locked in requirements.txt. Managed services sometimes force version upgrades that can change vector index behavior. The tradeoff is that we own OS patching and backup automation, but those are solvable with SSM + AWS Backup.
Q: How would you scale this platform if curry_bowl_express expanded to 120 locations instead of 12?
Several layers change at different thresholds. At 120 locations, the first bottleneck is the alert_detector DuckDB query: it's scanning 120 × 14 days of Gold Parquet which is 10× more data. Fix: pre-compute a fct_location_alerts dbt model that runs hourly and materialises the alert state — the alert_detector queries this materialised table instead of raw Parquet, reducing scan time from ~30s to under 1s. Second, the Qdrant collection grows 10× in vector count; at 500K vectors the existing m6i.large (8GB RAM) is still adequate but we'd add a replica for read scaling. Third, MSK Serverless auto-scales with the 10× volume increase; no change needed. Fourth, ECS Fargate auto-scaling at 10× traffic with the same policy, just more tasks. The architecture is designed for horizontal scale from day one — stateless compute, data in S3/Qdrant/MSK.
